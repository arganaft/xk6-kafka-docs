Accept JSON requests from the frontend, containing the required groupings and aggregate functions (from one or multiple data sources).

Guarantee that the delay from the subscription request to the delivery of the first snapshot does not exceed 100 ms.

A single dashboard may require data from different sources (joins or combination of results).

Key accepted decision: Apache Pinot is used as the OLAP store for fast analytical queries. Pinot efficiently handles aggregate queries with groupings, supports SQL, and can ingest real-time Kafka streams (including upsert semantics). However, Pinot cannot push updates to subscribers — it only responds to pull requests (HTTP/gRPC).

Current architecture draft and its weak points:

Service A (REST API) receives the JSON from the frontend, validates it using Apache Calcite, converts it into a SQL query targeting Pinot, and stores that SQL in Redis (keyed by subscription/user identifier).

Service B establishes WebSocket connections with the frontend. Service B reads the relevant SQL queries for the connected user from Redis and periodically polls Pinot, forwarding results to the frontend.

For optimization, a proxy service in front of Pinot is proposed to:

Merge identical or similar SQL queries to reduce load on Pinot.

Possibly receive data change notifications (mechanism undefined) and trigger polling only when needed.

Distribute queries evenly over time to smooth out spikes.

Open issues: unclear how to combine data from multiple sources (joins inside Pinot? if so, how to organize schema and data flow; if not, where and how to merge results). Polling creates a trade-off between latency and load, especially at millions of subscriptions.

Requirements and constraints:

Expected total incoming event rate: millions of events per second.

Data volume stored in Pinot: several terabytes (retention depth: weeks/months).

Occasional loss of individual events and duplication is acceptable, but the data visible to traders must be consistent (e.g., no “jumps” in aggregates due to uncoordinated updates).

The entire system will run on Kubernetes/Istio. All components must scale horizontally.

Technologies considered but rejected for the subscription use case:

RisingWave: limits on materialized views (max 256), extremely slow creation of new views (minutes/days).

Flink, Kafka Streams, ksqlDB: did not meet certain requirements (likely unable to serve point-read queries with low latency at the required scale of dynamic subscriptions).

Pinot was chosen for its ability to quickly execute aggregate queries and ease of scaling.

# Индексы, которые уже существуют

## Суть проблемы

Вы добавляете changeset с `createIndex`, но индекс уже есть в БД (создан вручную, другим скриптом, или из предыдущей миграции). Liquibase упадёт с ошибкой:

```
Index "idx_orders_user_id" already exists
```

---

## Способы решения

### 1. `preConditions` — самый правильный подход

```yaml
- changeSet:
    id: create-index-orders-user-id
    author: dev
    preConditions:
      - onFail: MARK_RAN
      - not:
          - indexExists:
              indexName: idx_orders_user_id
              tableName: orders
    changes:
      - createIndex:
          tableName: orders
          indexName: idx_orders_user_id
          columns:
            - column:
                name: user_id
    rollback:
      - dropIndex:
          tableName: orders
          indexName: idx_orders_user_id
```

**Что происходит:**
- Индекса нет → создаёт, записывает в `DATABASECHANGELOG` как `EXECUTED`
- Индекс есть → пропускает, записывает как `MARK_RAN`

### 2. SQL с `IF NOT EXISTS` (для PostgreSQL)

```yaml
- changeSet:
    id: create-index-orders-user-id
    author: dev
    changes:
      - sql:
          sql: >
            CREATE INDEX IF NOT EXISTS idx_orders_user_id 
            ON orders (user_id);
    rollback:
      - sql:
          sql: DROP INDEX IF EXISTS idx_orders_user_id;
```

**Просто, но:**
- Привязка к конкретной СУБД (`IF NOT EXISTS` есть не везде)
- Теряете абстракцию Liquibase

### 3. `failOnError: false` — грубый способ

```yaml
- changeSet:
    id: create-index-orders-user-id
    author: dev
    failOnError: false
    changes:
      - createIndex:
          tableName: orders
          indexName: idx_orders_user_id
          columns:
            - column:
                name: user_id
```

> ⚠️ **Не рекомендую.** Это подавит ЛЮБУЮ ошибку, не только «уже существует». Можете пропустить реальную проблему.

### 4. `changeLogSync` — для уже существующих баз

Если вы внедряете Liquibase в проект, где БД уже существует со всеми таблицами и индексами:

```bash
# Помечает ВСЕ changeSet как выполненные, не запуская их
liquibase changelogSync
```

Или точечно:

```bash
# До конкретного changeset
liquibase changelogSyncToTag --tag=v1.0.0
```

## Типичные сценарии и что делать

### Сценарий 1: Внедряете Liquibase в существующий проект

```
БД уже есть → таблицы есть → индексы есть
```

**Решение:** Создаёте «базовый» changelog, описывающий текущее состояние, и делаете `changelogSync`:

```yaml
# v0.0/001-baseline.yaml — описание того, что уже есть
- changeSet:
    id: baseline-create-users
    author: dev
    preConditions:
      - onFail: MARK_RAN
      - not:
          - tableExists:
              tableName: users
    changes:
      - createTable:
          tableName: users
          # ...

- changeSet:
    id: baseline-index-users-email
    author: dev
    preConditions:
      - onFail: MARK_RAN
      - not:
          - indexExists:
              indexName: idx_users_email
    changes:
      - createIndex:
          tableName: users
          indexName: idx_users_email
          columns:
            - column:
                name: email
```

Теперь этот changelog безопасен:
- **Пустая БД** → создаст всё с нуля
- **Существующая БД** → пропустит всё, пометит `MARK_RAN`

### Сценарий 2: Индекс создали вручную на проде

```
Changeset ещё не запускался, но DBA уже создал индекс руками
```

**Решение:** `preConditions` с `onFail: MARK_RAN` (способ 1).

### Сценарий 3: Переносите индекс из одного changeset в другой

```
Раньше индекс создавался в changeset с таблицей,
теперь хотите вынести в отдельный файл
```

> ⚠️ **Нельзя менять уже выполненный changeset!** Liquibase проверяет контрольные суммы.

**Решение:**

```yaml
# Новый changeset — только для сред, где индекса ещё нет
- changeSet:
    id: extract-index-orders-user-id
    author: dev
    preConditions:
      - onFail: MARK_RAN
      - not:
          - indexExists:
              indexName: idx_orders_user_id
    changes:
      - createIndex:
          tableName: orders
          indexName: idx_orders_user_id
          columns:
            - column:
                name: user_id
```

## Сводка: что когда использовать

```
┌─────────────────────────────────┬──────────────────────────┐
│ Ситуация                        │ Решение                  │
├─────────────────────────────────┼──────────────────────────┤
│ Новый проект, чистая БД         │ Просто createIndex       │
│ Индекс может существовать       │ preConditions + MARK_RAN │
│ Только PostgreSQL               │ IF NOT EXISTS            │
│ Внедрение в существующую БД     │ baseline + changelogSync │
│ Не знаю, что пошло не так       │ НЕ failOnError: false    │
└─────────────────────────────────┴──────────────────────────┘
```

**Золотое правило:** всегда используйте `preConditions` с проверкой существования. Это делает ваши миграции **идемпотентными** — безопасными для повторного запуска на любом окружении в любом состоянии.

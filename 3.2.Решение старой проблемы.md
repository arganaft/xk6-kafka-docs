# 1. Почему debug-страница “лечит” WebSocket

Вы нашли очень важную деталь:

> При входе на отладочную страницу запрашивается сертификат. Если выбрать сертификат — после этого WS работает.

Это почти полностью объясняет поведение.

Скорее всего схема такая:

```text
1. Frontend пытается открыть wss://websocket-istio-test-k8s.../ws
2. Istio Gateway / внешний Envoy / корпоративный proxy требует клиентский сертификат.
3. Браузер не показывает выбор сертификата для WebSocket subresource
   или STOMP-клиент сам отменяет handshake по timeout.
4. WebSocket handshake отменяется.
5. Вы открываете debug-страницу как обычную top-level HTTPS navigation.
6. Браузер показывает окно выбора клиентского сертификата.
7. Вы выбираете сертификат.
8. Браузер запоминает выбор сертификата для этого host/SNI.
9. Следующий wss://.../ws уже проходит.
```

То есть debug-страница не “прогревает Spring WebSocket”. Она случайно выполняет роль:

```text
certificate bootstrap / TLS client-auth initialization
```

# 2. Браузер не “подставляет сертификат в каждое сообщение”

Вы написали:

> вероятно браузер при каждом сообщении подставляет сертификат для Istio envoy

Не совсем.

Клиентский сертификат используется **во время TLS handshake**. Для `wss://` это происходит до WebSocket-протокола.

Упрощённо:

```text
TCP connection
TLS handshake
client certificate selection/authentication
HTTP GET /ws Upgrade: websocket
HTTP 101 Switching Protocols
WebSocket frames
STOMP frames
```

После того как TLS-соединение установлено, сертификат не отправляется “в каждом WebSocket-сообщении”. WebSocket/STOMP-сообщения уже идут внутри установленного TLS-канала.

# 4. Очень важная деталь про `@stomp/stompjs` и `connectionTimeout`

В вашем minified-коде из `@stomp/stompjs` есть вот это:

```javascript
this.connectionTimeout > 0 && (
  this._connectionWatcher = setTimeout(() => {
    this.connected || (
      this.debug(`Connection not established in ${this.connectionTimeout}ms, closing socket`),
      this.forceDisconnect()
    )
  }, this.connectionTimeout)
)
```

То есть если у STOMP-клиента задан `connectionTimeout`, например `5000` ms, то может происходить такое:

```text
1. Открывается wss://.../ws.
2. Браузер/Envoy начинает TLS client certificate flow.
3. Пользователь ещё выбирает сертификат или браузер не показывает prompt.
4. @stomp/stompjs не дождался CONNECTED.
5. @stomp/stompjs сам закрывает WebSocket.
6. Chrome пишет: WebSocket opening handshake was canceled.
```

Поэтому для проверки я бы попросил фронтендера временно сделать так:

```typescript
const client = new Client({
  brokerURL: 'wss://websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru/ws',

  // Временно для диагностики:
  connectionTimeout: 0,

  reconnectDelay: 5000,

  debug: (message) => {
    console.log('[STOMP]', message);
  },

  onConnect: (frame) => {
    console.log('STOMP connected', frame);
  },

  onWebSocketError: (event) => {
    console.error('WebSocket error', event);
  },

  onWebSocketClose: (event) => {
    console.warn(
      'WebSocket closed',
      'code=', event.code,
      'reason=', event.reason,
      'wasClean=', event.wasClean
    );
  },

  onStompError: (frame) => {
    console.error('STOMP error', frame.headers['message']);
    console.error(frame.body);
  }
});

client.activate();
```

Если после `connectionTimeout: 0` поведение изменится, значит STOMP-клиент действительно сам отменял handshake, пока браузер/сертификат не успевали завершить TLS.

Для production можно поставить не `0`, а большой timeout, например `60000`.

# 8. Если frontend хочет получить `sessionId` в первом STOMP-сообщении

Тогда ваш backend можно доработать так.

```java
package com.pivotService.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.messaging.Message;
import org.springframework.messaging.MessageChannel;
import org.springframework.messaging.simp.SimpMessageHeaderAccessor;
import org.springframework.messaging.simp.SimpMessageType;
import org.springframework.messaging.simp.config.ChannelRegistration;
import org.springframework.messaging.simp.config.MessageBrokerRegistry;
import org.springframework.messaging.simp.stomp.StompHeaderAccessor;
import org.springframework.messaging.support.ChannelInterceptor;
import org.springframework.messaging.support.MessageBuilder;
import org.springframework.scheduling.concurrent.ThreadPoolTaskScheduler;
import org.springframework.web.socket.config.annotation.EnableWebSocketMessageBroker;
import org.springframework.web.socket.config.annotation.StompEndpointRegistry;
import org.springframework.web.socket.config.annotation.WebSocketMessageBrokerConfigurer;

@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    public static final String CONNECTED_SESSION_ID_HEADER = "x-simp-session-id";

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("*");
    }

    @Override
    public void configureClientOutboundChannel(ChannelRegistration registration) {
        registration.interceptors(new ChannelInterceptor() {

            @Override
            public Message<?> preSend(Message<?> message, MessageChannel channel) {
                StompHeaderAccessor accessor = StompHeaderAccessor.wrap(message);

                if (!SimpMessageType.CONNECT_ACK.equals(accessor.getMessageType())) {
                    return message;
                }

                String sessionId = accessor.getSessionId();

                if (sessionId == null) {
                    sessionId = (String) message.getHeaders()
                            .get(SimpMessageHeaderAccessor.SESSION_ID_HEADER);
                }

                if (sessionId != null) {
                    accessor.setNativeHeader(CONNECTED_SESSION_ID_HEADER, sessionId);
                }

                return MessageBuilder.createMessage(
                        message.getPayload(),
                        accessor.getMessageHeaders()
                );
            }
        });
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/topic", "/queue")
                .setTaskScheduler(webSocketHeartbeatScheduler())
                .setHeartbeatValue(new long[]{10000, 10000});

        registry.setApplicationDestinationPrefixes("/app");
        registry.setUserDestinationPrefix("/user");
    }

    @Bean
    public ThreadPoolTaskScheduler webSocketHeartbeatScheduler() {
        ThreadPoolTaskScheduler taskScheduler = new ThreadPoolTaskScheduler();
        taskScheduler.setPoolSize(1);
        taskScheduler.setThreadNamePrefix("heartbeat-");
        taskScheduler.initialize();
        return taskScheduler;
    }
}
```

Frontend будет читать так:

```typescript
onConnect: (frame) => {
  const sessionId = frame.headers['x-simp-session-id'];
  console.log('sessionId from server:', sessionId);
}
```

Важно: это **не HTTP header**. Это STOMP header в frame `CONNECTED`.

Если там:

```yaml

```

или:

```yaml

  mode: OPTIONAL_MUTUAL
```

то Gateway запрашивает клиентский сертификат у браузера.

Если WebSocket должен работать из SPA без отдельного шага выбора сертификата, нужно одно из решений:

## Вариант A. Не требовать client certificate на WebSocket host

Например:

```yaml
tls:
  mode: SIMPLE
```

А авторизацию делать через JWT/cookie/token/STOMP headers.

## Вариант B. Разместить WebSocket под тем же origin, что и frontend

Например, вместо:

```text
Frontend: https://simulation-ui.apps.a8uaat10.k8s.delta.sbrf.ru
WebSocket: wss://websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru/ws
```

сделать:

```text
Frontend: https://simulation-ui.apps.a8uaat10.k8s.delta.sbrf.ru
WebSocket: wss://simulation-ui.apps.a8uaat10.k8s.delta.sbrf.ru/api/pivot/ws
```

Тогда сертификат/auth/session будут инициализироваться при открытии самого frontend-приложения.

Это часто лучший вариант для браузерных приложений.

## Вариант C. Сделать явный cert-init endpoint

Если требования безопасности говорят, что client certificate обязателен, тогда debug-страницу надо заменить на нормальный production endpoint, например:

```text
https://websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru/v2/auth/cert-init
```

Он может возвращать простую страницу или `204`, но открываться должен как top-level navigation/popup, чтобы браузер показал выбор сертификата.

Например:

```java
@GetMapping("/v2/auth/cert-init")
@ResponseStatus(HttpStatus.NO_CONTENT)
public void certInit() {
}
```

Но важно: это workaround. Он нужен не для WebSocket, а для TLS client certificate flow.

# 13. Важный архитектурный момент про несколько POD-ов

У вас включён:

```java
registry.enableSimpleBroker("/topic", "/queue")
```

Это in-memory broker внутри каждого POD-а.

То есть если у вас несколько реплик, то:

```text
POD A знает только свои WebSocket-сессии.
POD B знает только свои WebSocket-сессии.
```

Если WebSocket клиента открыт на `POD A`, а REST/POST-запрос попал на `POD B`, то `POD B` может не знать эту сессию.

Поэтому есть два нормальных варианта.

## Вариант 1. Осознанно использовать sticky sessions

Тогда нужно правильно настроить sticky routing на уровне Istio/Ingress, чтобы REST и WS одного клиента попадали в один POD.

Но это хуже переживает rolling update и падение POD-а.

## Вариант 2. Использовать внешний broker

Например:

```java
registry.enableStompBrokerRelay("/topic", "/queue")
        .setRelayHost("rabbitmq")
        .setRelayPort(61613)
        .setClientLogin("guest")
        .setClientPasscode("guest");
```

Тогда сообщения не завязаны на in-memory broker конкретного POD-а.

Для production с несколькими репликами это обычно более правильный путь.

## Шаг 1. Подтвердить, что запрос не доходит до Spring

Добавьте `HandshakeInterceptor`.

```java
package com.pivotService.config;

import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.http.HttpHeaders;
import org.springframework.http.server.ServerHttpRequest;
import org.springframework.http.server.ServerHttpResponse;
import org.springframework.stereotype.Component;
import org.springframework.web.socket.WebSocketHandler;
import org.springframework.web.socket.server.HandshakeInterceptor;

import java.util.Map;

@Component
@Slf4j
public class WebSocketHandshakeLoggingInterceptor implements HandshakeInterceptor {

    @Value("${HOSTNAME:local}")
    private String podName;

    @Override
    public boolean beforeHandshake(
            ServerHttpRequest request,
            ServerHttpResponse response,
            WebSocketHandler wsHandler,
            Map<String, Object> attributes
    ) {
        HttpHeaders headers = request.getHeaders();

        log.info(
                "WS HANDSHAKE BEFORE: pod={}, uri={}, origin={}, cookie={}, upgrade={}, connection={}, protocol={}, xfcc={}",
                podName,
                request.getURI(),
                headers.getFirst(HttpHeaders.ORIGIN),
                headers.getFirst(HttpHeaders.COOKIE),
                headers.getFirst(HttpHeaders.UPGRADE),
                headers.getFirst(HttpHeaders.CONNECTION),
                headers.getFirst("Sec-WebSocket-Protocol"),
                headers.getFirst("x-forwarded-client-cert")
        );

        return true;
    }

    @Override
    public void afterHandshake(
            ServerHttpRequest request,
            ServerHttpResponse response,
            WebSocketHandler wsHandler,
            Exception exception
    ) {
        if (exception == null) {
            log.info("WS HANDSHAKE AFTER: pod={}, uri={}, status=success", podName, request.getURI());
        } else {
            log.warn("WS HANDSHAKE AFTER: pod={}, uri={}, status=failed", podName, request.getURI(), exception);
        }
    }
}
```

Подключите:

```java
@Override
public void registerStompEndpoints(StompEndpointRegistry registry) {
    registry.addEndpoint("/ws")
            .addInterceptors(webSocketHandshakeLoggingInterceptor)
            .setAllowedOriginPatterns("*");
}
```

```text
Istio Gateway / внешний Envoy требует client certificate для host 
websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru.

Когда frontend пытается открыть wss://.../ws как subresource,
браузер не проходит интерактивный выбор сертификата или STOMP-клиент
закрывает handshake по timeout.

Когда вы открываете debug-страницу как обычную HTTPS-страницу,
браузер показывает certificate prompt, сертификат выбирается,
после чего wss://.../ws начинает работать.
```

Это **не лечится изменением Spring WebSocketConfig**.

Spring-конфиг можно улучшить:

- убрать `@EnableWebSocket`;
- добавить `x-simp-session-id` в STOMP `CONNECTED`;
- добавить handshake-логи;
- использовать внешний broker или осознанную sticky session при нескольких POD-ах.

Но главный фикс — в одном из мест:

```text
Istio Gateway TLS config
client certificate auth flow
frontend origin / routing architecture
STOMP client timeout
sticky/session affinity
```

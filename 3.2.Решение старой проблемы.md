Да, звучит странно, но по описанию уже есть сильный подозреваемый: **cookie/session affinity/sticky session на уровне ingress/Istio/Spring Session/Security** либо **разные POD-ы ведут себя не одинаково**.

А предложенный класс с `x-simp-session-id` — **в целом правильнее**, чем ваш старый вариант с JSON body в `CONNECTED` frame. Но важно: **это не починит проблему “нужно сначала открыть debug-страницу”**. Это отдельная доработка протокола STOMP.

# 1. По `sessionId` в первом сообщении

Ваш старый код делал плохо:

```java
return MessageBuilder.createMessage(
        obje,
        message.getHeaders()
);
```

Вы подменяли payload системного STOMP-сообщения подключения. Так делать не стоит.

Предложенный вариант лучше:

```java
accessor.setNativeHeader(CONNECTED_SESSION_ID_HEADER, sessionId);
```

То есть вы добавляете STOMP header в `CONNECTED` frame.

Это нормальная идея.

STOMP frame после подключения будет примерно такой:

```text
CONNECTED
version:1.2
heart-beat:10000,10000
x-simp-session-id:abc123

^@
```

Frontend сможет прочитать это в `@stomp/stompjs` так:

```javascript
const client = new Client({
  brokerURL: 'wss://websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru/ws',

  onConnect: (frame) => {
    const sessionId = frame.headers['x-simp-session-id'];

    console.log('Connected. Spring sessionId:', sessionId);
  }
});
```

Важно: это **не HTTP header**, а **STOMP header**. Его надо читать из `frame.headers`, а не из Network request/response headers браузера.

# 2. Я бы немного поправил ваш `WebSocketConfig`

Я бы убрал `@EnableWebSocket`. Для STOMP over WebSocket достаточно:

```java
@EnableWebSocketMessageBroker
```

`@EnableWebSocket` нужен для более низкоуровневого raw WebSocket API. В вашем случае он лишний.

Пример:

```java
package com.pivotService.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.messaging.Message;
import org.springframework.messaging.MessageChannel;
import org.springframework.messaging.simp.SimpMessageType;
import org.springframework.messaging.simp.config.ChannelRegistration;
import org.springframework.messaging.simp.config.MessageBrokerRegistry;
import org.springframework.messaging.simp.stomp.StompHeaderAccessor;
import org.springframework.messaging.support.ChannelInterceptor;
import org.springframework.messaging.support.MessageBuilder;
import org.springframework.scheduling.concurrent.ThreadPoolTaskScheduler;
import org.springframework.web.socket.config.annotation.EnableWebSocketMessageBroker;
import org.springframework.web.socket.config.annotation.StompEndpointRegistry;
import org.springframework.web.socket.config.annotation.WebSocketMessageBrokerConfigurer;

@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    public static final String CONNECTED_SESSION_ID_HEADER = "x-simp-session-id";

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("*");
    }

    @Override
    public void configureClientOutboundChannel(ChannelRegistration registration) {
        registration.interceptors(new ChannelInterceptor() {

            @Override
            public Message<?> preSend(Message<?> message, MessageChannel channel) {
                StompHeaderAccessor accessor = StompHeaderAccessor.wrap(message);

                if (!SimpMessageType.CONNECT_ACK.equals(accessor.getMessageType())) {
                    return message;
                }

                String sessionId = accessor.getSessionId();

                if (sessionId != null) {
                    accessor.setNativeHeader(CONNECTED_SESSION_ID_HEADER, sessionId);
                }

                return MessageBuilder.createMessage(
                        message.getPayload(),
                        accessor.getMessageHeaders()
                );
            }
        });
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/topic", "/queue")
                .setTaskScheduler(webSocketHeartbeatScheduler())
                .setHeartbeatValue(new long[]{10000, 10000});

        registry.setApplicationDestinationPrefixes("/app");
        registry.setUserDestinationPrefix("/user");
    }

    @Bean
    public ThreadPoolTaskScheduler webSocketHeartbeatScheduler() {
        ThreadPoolTaskScheduler taskScheduler = new ThreadPoolTaskScheduler();
        taskScheduler.setPoolSize(1);
        taskScheduler.setThreadNamePrefix("heartbeat-");
        taskScheduler.initialize();
        return taskScheduler;
    }
}
```

# 3. Но это не объясняет проблему с debug-страницей

Вот эта часть наиболее важная:

> если открыть страницу локально и попробовать подключиться к серверу, связь не устанавливалась, но если запросить страницу с сервера, то связь с этим POD устанавливалась, причём в логах видно, что websocket связь устанавливается с тем pod, с которого была отдана страница.

Это очень похоже не на проблему Java-кода WebSocket, а на одну из этих проблем:

1. **ingress/Istio выставляет sticky-cookie;**
2. **Spring/Security выставляет `JSESSIONID`;**
3. **после открытия страницы браузер получает cookie для домена websocket-сервиса;**
4. **следующий WebSocket-запрос идёт с этой cookie;**
5. **load balancer направляет WebSocket на тот же POD;**
6. **без cookie запрос попадает на другой POD или в другой route/subset и там ломается.**

То есть debug-страница, скорее всего, не “прогревает WebSocket”. Она побочно создаёт состояние.

Это нормально.

должен завершаться ошибкой.

Поэтому вот это:

```text

```

само по себе не баг.

Но вот это должно работать:

```text
wss://websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru/ws
```

с ответом:

```text
101 Switching Protocols
```

# 6. Почему Insomnia работает, а браузер нет

Это тоже важная подсказка.

Insomnia и браузер отличаются.

Браузер почти всегда отправляет:

```http
Origin: https://simulation-ui.apps.a8uaat10.k8s.delta.sbrf.ru
```

Insomnia может не отправлять `Origin` вообще.

Браузер работает с cookies, SameSite, CORS-политиками, корпоративными политиками браузера и т.д.

Insomnia — нет или делает это иначе.

Поэтому ситуация:

```text
Insomnia подключается по wss.
Браузер не подключается.
```

часто означает проблему в одном из мест:

- `Origin`;
- cookies;
- Spring Security;
- CSRF;
- sticky sessions;
- ingress/Istio routing;
- разные версии приложения в разных POD-ах;
- TLS/corporate browser policy.

# 7. Самая вероятная причина в вашем описании

С учётом фразы:

> websocket связь устанавливается с тем pod, с которого была отдана страница

самая вероятная причина:

```text
При открытии debug-страницы выставляется cookie, которая включает sticky routing.
После этого WebSocket попадает в тот же POD.
Без этой cookie WebSocket попадает в другой POD или другой route и ломается.
```

Это может быть cookie от:

- `ingress-nginx`;
- `Istio`;
- Spring Boot;
- Spring Security;
- Spring Session;
- корпоративного reverse proxy перед Kubernetes.

Проверьте `DestinationRule`.

Ищите `consistentHash`:

# 11. Проверьте, что все POD-ы одинаковые

Очень частая причина таких “мистических” проблем:

```text
часть POD-ов запущена со старой версией;
часть POD-ов имеет другой config;
часть POD-ов не принимает /ws;
часть POD-ов не готова, но всё ещё получает трафик;
```

Проверьте:

```bash
kubectl get pods -n <namespace> -l app=<app-label> -o wide
```

Проверьте image у всех POD-ов:

```bash
kubectl get pods -n <namespace> -l app=<app-label> \
```

Если в endpoints есть POD, который на самом деле не умеет нормально принимать WebSocket, его надо убрать из балансировки.

# 12. Добавьте логирование handshake

Чтобы перестать гадать, добавьте `HandshakeInterceptor`.

```java
package com.pivotService.config;

import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.http.HttpHeaders;
import org.springframework.http.server.ServerHttpRequest;
import org.springframework.http.server.ServerHttpResponse;
import org.springframework.stereotype.Component;
import org.springframework.web.socket.WebSocketHandler;
import org.springframework.web.socket.server.HandshakeInterceptor;

import java.util.Map;

@Component
@Slf4j
public class WebSocketHandshakeLoggingInterceptor implements HandshakeInterceptor {

    @Value("${HOSTNAME:local}")
    private String podName;

    @Override
    public boolean beforeHandshake(
            ServerHttpRequest request,
            ServerHttpResponse response,
            WebSocketHandler wsHandler,
            Map<String, Object> attributes
    ) {
        HttpHeaders headers = request.getHeaders();

        log.info(
                "WS HANDSHAKE BEFORE: pod={}, uri={}, origin={}, cookie={}, upgrade={}, connection={}, secWebSocketProtocol={}",
                podName,
                request.getURI(),
                headers.getFirst(HttpHeaders.ORIGIN),
                headers.getFirst(HttpHeaders.COOKIE),
                headers.getFirst(HttpHeaders.UPGRADE),
                headers.getFirst(HttpHeaders.CONNECTION),
                headers.getFirst("Sec-WebSocket-Protocol")
        );

        return true;
    }

    @Override
    public void afterHandshake(
            ServerHttpRequest request,
            ServerHttpResponse response,
            WebSocketHandler wsHandler,
            Exception exception
    ) {
        if (exception == null) {
            log.info("WS HANDSHAKE AFTER: pod={}, uri={}, status=success", podName, request.getURI());
        } else {
            log.warn("WS HANDSHAKE AFTER: pod={}, uri={}, status=failed", podName, request.getURI(), exception);
        }
    }
}
```

Подключите его:

```java
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    private final WebSocketHandshakeLoggingInterceptor handshakeLoggingInterceptor;

    public WebSocketConfig(WebSocketHandshakeLoggingInterceptor handshakeLoggingInterceptor) {
        this.handshakeLoggingInterceptor = handshakeLoggingInterceptor;
    }

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .addInterceptors(handshakeLoggingInterceptor)
                .setAllowedOriginPatterns("*");
    }

    // остальные методы
}
```

После этого сравните логи:

## Сценарий А

Без открытия debug-страницы:

```text
frontend -> wss://.../ws
```

## Сценарий Б

После открытия:

```text
https://.../websocket/trades
frontend -> wss://.../ws
```

В логах надо сравнить:

```text
pod
origin
cookie
uri
exception
```

Если в сценарии Б появляется cookie — почти точно дело в cookie/sticky/session.

# 13. Добавьте логирование STOMP `CONNECT`

Handshake может пройти, а STOMP может упасть позже. Поэтому добавьте inbound interceptor.

```java
package com.pivotService.config;

import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.messaging.Message;
import org.springframework.messaging.MessageChannel;
import org.springframework.messaging.simp.stomp.StompCommand;
import org.springframework.messaging.simp.stomp.StompHeaderAccessor;
import org.springframework.messaging.support.ChannelInterceptor;
import org.springframework.messaging.support.MessageHeaderAccessor;
import org.springframework.stereotype.Component;

@Component
@Slf4j
public class StompConnectLoggingInterceptor implements ChannelInterceptor {

    @Value("${HOSTNAME:local}")
    private String podName;

    @Override
    public Message<?> preSend(Message<?> message, MessageChannel channel) {
        StompHeaderAccessor accessor = MessageHeaderAccessor.getAccessor(
                message,
                StompHeaderAccessor.class
        );

        if (accessor == null) {
            return message;
        }

        if (StompCommand.CONNECT.equals(accessor.getCommand())) {
            log.info(
                    "STOMP CONNECT: pod={}, sessionId={}, nativeHeaders={}",
                    podName,
                    accessor.getSessionId(),
                    accessor.toNativeHeaderMap()
            );
        }

        if (StompCommand.SUBSCRIBE.equals(accessor.getCommand())) {
            log.info(
                    "STOMP SUBSCRIBE: pod={}, sessionId={}, destination={}",
                    podName,
                    accessor.getSessionId(),
                    accessor.getDestination()
            );
        }

        if (StompCommand.SEND.equals(accessor.getCommand())) {
            log.info(
                    "STOMP SEND: pod={}, sessionId={}, destination={}",
                    podName,
                    accessor.getSessionId(),
                    accessor.getDestination()
            );
        }

        return message;
    }
}
```

Регистрация:

```java
@Override
public void configureClientInboundChannel(ChannelRegistration registration) {
    registration.interceptors(stompConnectLoggingInterceptor);
}
```

# 15. Как фронтенду читать `sessionId`

Для `@stomp/stompjs`:

```javascript
import { Client } from '@stomp/stompjs';

const client = new Client({
  brokerURL: 'wss://websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru/ws',

  debug: (str) => {
    console.log('[STOMP]', str);
  },

  onConnect: (frame) => {
    console.log('STOMP connected frame:', frame);

    const sessionId = frame.headers['x-simp-session-id'];

    if (!sessionId) {
      console.warn('x-simp-session-id header not found');
    } else {
      console.log('Spring simpSessionId:', sessionId);
    }

    client.subscribe('/user/topic/trades', (message) => {
      console.log('Message:', message.body);
    });
  },

  onWebSocketError: (event) => {
    console.error('WebSocket error:', event);
  },

  onWebSocketClose: (event) => {
    console.warn(
      'WebSocket closed:',
      'code=', event.code,
      'reason=', event.reason,
      'wasClean=', event.wasClean
    );
  },

  onStompError: (frame) => {
    console.error('STOMP error:', frame.headers['message']);
    console.error(frame.body);
  }
});

client.activate();
```

# 16. Важное замечание про `sessionId`

Я бы всё-таки уточнил у фронтендера, зачем ему нужен именно server-side `simpSessionId`.

`sessionId`:

- меняется при каждом переподключении;
- не является стабильным идентификатором клиента;
- не является секретом;
- не должен использоваться как бизнес-идентификатор;
- не должен использоваться как авторизация.

Если фронту нужно идентифицировать себя перед backend, правильнее передавать `clientId`:

```javascript
const client = new Client({
  brokerURL: 'wss://websocket-istio-test-k8s.quantara-dev.delta.sbrf.ru/ws',

  connectHeaders: {
    clientId: 'client-123'
  },

  onConnect: (frame) => {
    const serverSessionId = frame.headers['x-simp-session-id'];
  }
});
```

А backend может прочитать `clientId` на STOMP `CONNECT`.

# 17. Главный вывод

Предложенное изменение с:

```java
accessor.setNativeHeader("x-simp-session-id", sessionId);
```

— нормальное. Его можно сделать.

Но оно решает только задачу:

```text
дать frontend sessionId в STOMP CONNECTED frame
```

Оно **не объясняет и не чинит** проблему:

```text
WebSocket начинает работать только после открытия debug-страницы с сервера.
```

Эта проблема почти наверняка в одном из мест:

```text
cookie
sticky session
JSESSIONID
Spring Security session
CSRF
ingress-nginx affinity
Istio consistentHash
разные версии POD-ов
неодинаковая готовность POD-ов
маршрутизация /ws в разные upstream/subset
```

Первое, что я бы сделал:

1. Открыл бы debug-страницу.
2. Посмотрел бы её `Response Headers`.
3. Нашёл бы `Set-Cookie`.
4. Посмотрел бы последующий `wss://.../ws`.
5. Проверил бы, появился ли там `Cookie`.
6. По cookie понял бы, кто именно создаёт состояние: Spring, ingress или Istio.

Если после открытия страницы появляется cookie, то это уже не “магия WebSocket”, а обычное состояние браузера/балансировщика. Debug-страницу надо убирать из flow, а причину — чинить явно: либо сделать все POD-ы одинаковыми и stateless для handshake, либо настроить sticky session осознанно, либо вынести session state во внешний storage.

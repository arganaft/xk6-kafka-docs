Глядя на ваши цифры, сразу бросается в глаза: на новой базе настройки **масштабированы пропорционально росту оперативной памяти** (примерно в 3 раза больше, чем на старой). В целом, они выглядят адекватно для 48 ГБ RAM, но есть нюансы.

Да, вы можете «подкрутить» параметры через SQL, но важно понимать, какие из них вступят в силу сразу, а какие потребуют перезагрузки всей службы PostgreSQL.

---

### 1. Как менять параметры через SQL

В современных версиях Postgres (и Pangolin) используется команда `ALTER SYSTEM`. Она записывает настройки в специальный файл `postgresql.auto.conf`, который имеет приоритет над основным конфигом.

#### Группа А: Требуют перезагрузки (Restart required)

Такие параметры, как `shared_buffers` и `max_connections`, нельзя изменить «на лету», так как они выделяют память при старте процесса.

```sql
ALTER SYSTEM SET shared_buffers = '12GB'; -- 25% от 48GB
ALTER SYSTEM SET max_connections = 500;   -- См. примечание ниже
-- После этого НУЖНО перезапустить сервис через консоль сервера (systemctl restart ...)

```

#### Группа Б: Достаточно «перечитать» конфиг (Reload)

Эти параметры можно применить без остановки базы.

```sql
ALTER SYSTEM SET work_mem = '64MB';
ALTER SYSTEM SET random_page_cost = 1.1; -- ОЧЕНЬ важно для SSD
-- После этого выполните команду для применения:
SELECT pg_reload_conf();

```

### 2. Что именно я советую изменить прямо сейчас

Судя по вашим данным, новая база работает медленнее не из-за нехватки памяти, а, скорее всего, из-за **логики планировщика** или **конкуренции ресурсов**.

#### 1. Уменьшите `max_connections`

У вас стоит **2000**. Для 16 ядер это **очень много**.

* **Проблема:** Каждый процесс потребляет память и создает нагрузку на переключение контекста процессора. Если у вас реально 2000 активных соединений, база будет «задыхаться».
* **Совет:** Если приложение позволяет, снизьте до 500 (или используйте PgBouncer).

#### 2. Проверьте `random_page_cost` (Главный подозреваемый)

В вашем списке его нет, но если он стоит по умолчанию (**4.0**), то Postgres думает, что у вас медленные HDD диски, и игнорирует индексы, предпочитая сканировать таблицы целиком.

* **Решение:**
```sql
ALTER SYSTEM SET random_page_cost = 1.1;
SELECT pg_reload_conf();

```



#### 3. `work_mem`

У вас стоит около 24 МБ. Это неплохо, но если запросы сложные (много Join и Sort), этого может быть мало, и данные пишутся во временные файлы на диск.

* **Попробуйте:** поднять до `64MB` или `128MB`.

### 3. Сравнение параметров (визуально)

Вот как ваши текущие настройки соотносятся с рекомендуемыми для 48 ГБ RAM:

| Параметр | У вас сейчас (приблизительно) | Рекомендуется для 48GB | Нужен рестарт? |
| --- | --- | --- | --- |
| **shared_buffers** | ~12 ГБ (ОК) | 12 ГБ | **Да** |
| **effective_cache_size** | ~36 ГБ (ОК) | 36 ГБ | Нет |

| **maintenance_work_mem** | ~4.8 ГБ (ОК) | 2 - 4 ГБ | Нет |

### Важный момент про Pangolin

Pangolin — это доработанный Postgres (часто от Ростелекома). В нем могут быть специфические параметры безопасности или аудита, которые включены по умолчанию и «едят» производительность на запись (например, расширенный логинг).

**Что сделать в DBeaver прямо сейчас:**
Запустите этот запрос, чтобы увидеть, не «тупит» ли база из-за ожиданий (locks):

```sql
SELECT pid, state, wait_event_type, wait_event, query 
FROM pg_stat_activity 
WHERE state != 'idle';

```

Если в колонке `wait_event` много значений типа `Lock`, значит, проблема не в скорости дисков или памяти, а в блокировках таблиц.

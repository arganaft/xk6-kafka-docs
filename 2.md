Да, универсальный генератор сделать можно. Но ключевая идея не в том, чтобы определить: **«таблица плоская или сложная»**, а в том, чтобы определить **зернистость результата**: одна строка результата — это сделка, cashflow-элемент или агрегированная группа.

## Короткий ответ

1. **Да, можно сделать один универсальный генератор**, но внутри он должен выбирать режим построения SQL:
   - обычный плоский режим;
   - режим «вернуть массивы как JSON-строки»;
   - режим `ARRAY JOIN`, когда массивы нужно превратить в строки.

2. **Для вашего примера с `rowGroupColumns`, `pivotColumns` и `sum(cashflows.paymentAmount)` нужен именно `ARRAY JOIN`**. Просто вернуть массивы строкой здесь нельзя, потому что агрегировать и группировать надо не массив целиком, а элементы массива.

3. **Не нужно определять “сложность” всей таблицы на каждый запрос.** Лучше хранить в метаданных полей признак: поле физически является `Array(...)` или нет. Эти метаданные можно загрузить один раз из `system.columns` ClickHouse и закешировать.

4. **Возвращать массивы как `['PAY','RECEIVE']` можно**, но это подходит только для режима отображения без группировок, pivot, агрегаций и фильтров по элементам массива. Лучше возвращать JSON-строку через `toJSONString(...)`, например `["PAY","RECEIVE"]`.

---

## Главная проблема в вашем SQL

Ваш текущий генератор делает так:

```sql
SELECT
    "portfolio_code" AS "Trade.portfolio.code",
    "cashflows.direction" AS "Trade.cashflows.direction",
    "cashflows.paymentCurrency_code" AS "Trade.cashflows.paymentCurrency.code",
    sum("cashflows.paymentAmount") AS "Trade.cashflows.paymentAmount"
FROM "default"."livebook322" FINAL
GROUP BY
    "portfolio_code",
    "cashflows.direction",
    "cashflows.paymentCurrency_code"
LIMIT 3
```

Но `cashflows.paymentAmount` — это не число, а массив:

```sql
Array(Nullable(Decimal(18, 6)))
```

То есть ClickHouse видит примерно такое:

```sql
sum([50.000000, 555075.000000])
```

А нужно агрегировать элементы массива:

```text
50.000000
555075.000000
...
```

Поэтому для таких запросов нужно сначала «развернуть» массивы через `ARRAY JOIN`.

## Правильная модель: 3 режима генерации

| Режим | Когда использовать | Что происходит с массивами | `Entity.id` |
|---|---|---|---|
| **Flat** | В запросе только обычные scalar-поля | Обычный `SELECT` | `id` из таблицы |
| **Arrays as JSON** | Нужно только показать массивы в ячейке, без группировок/агрегаций по ним | `toJSONString(arrayColumn)` | `id` из таблицы |
| **Array Join / Explode** | Нужно группировать, pivot-ить, фильтровать, сортировать или агрегировать элементы массива | `ARRAY JOIN` превращает массив в строки | synthetic id: `id + index` или hash/group key |

---

## Для вашего конкретного запроса нужен `ARRAY JOIN`

Ваш frontend присылает:

```json
{
  "rowGroupColumns": [
    { "fieldKey": "Trade.portfolio.code" },
    { "fieldKey": "Trade.cashflows.paymentDate" },
    { "fieldKey": "Trade.cashflows.counterparty.code" }
  ],
  "valueColumns": [
    { "fieldKey": "Trade.cashflows.paymentAmount", "aggFunc": "sum" }
  ],
  "pivotColumns": [
    { "fieldKey": "Trade.cashflows.direction" },
    { "fieldKey": "Trade.cashflows.paymentCurrency.code" }
  ]
}
```

Здесь `paymentAmount`, `direction`, `paymentCurrency.code` — элементы массива `cashflows`.

Правильный SQL для первого уровня группировки должен быть примерно таким:

```sql
SELECT
    concat(
        'g:',
        ifNull(toString(`portfolio_code`), 'NULL'),
        '|',
        ifNull(toString(__cf_direction), 'NULL'),
        '|',
        ifNull(toString(__cf_payment_currency), 'NULL')
    ) AS `Entity.id`,

    `portfolio_code` AS `Trade.portfolio.code`,
    __cf_direction AS `Trade.cashflows.direction`,
    __cf_payment_currency AS `Trade.cashflows.paymentCurrency.code`,
    sum(__cf_payment_amount) AS `Trade.cashflows.paymentAmount`

FROM `default`.`livebook322` FINAL

ARRAY JOIN
    `cashflows.direction` AS __cf_direction,
    `cashflows.paymentCurrency_code` AS __cf_payment_currency,
    `cashflows.paymentAmount` AS __cf_payment_amount

GROUP BY
    `portfolio_code`,
    __cf_direction,
    __cf_payment_currency

LIMIT 3
```

Важный момент: **не нужно добавлять `id` и `globalId` в `GROUP BY`**, если вы хотите получить агрегат по портфелю, направлению и валюте.

Ваш исправленный пример:

```sql
GROUP BY
    id,
    globalId,
    portfolio_code,
    `cashflows.direction`,
    `cashflows.paymentCurrency_code`
```

логически меняет результат. Вы группируете уже не по бизнес-ключам, а по каждой отдельной сделке. Поэтому суммы получаются на уровне конкретной строки таблицы, а не на уровне группы.

Для grouped rows `Entity.id` лучше делать синтетическим, например из значений группировки.

## Для detail-режима нужен другой `Entity.id`

Если вы хотите показать каждую cashflow-строку отдельно, тогда нужен индекс элемента массива:

```sql
SELECT
    concat(
        toString(ifNull(id, globalId)),
        ':cashflows:',
        toString(__cf_index)
    ) AS `Entity.id`,

    id AS `Trade.id`,
    globalId AS `Trade.globalId`,
    portfolio_code AS `Trade.portfolio.code`,

    __cf_direction AS `Trade.cashflows.direction`,
    __cf_payment_currency AS `Trade.cashflows.paymentCurrency.code`,
    __cf_payment_amount AS `Trade.cashflows.paymentAmount`

FROM `default`.`livebook322` FINAL

ARRAY JOIN
    arrayEnumerate(`cashflows.paymentAmount`) AS __cf_index,
    `cashflows.direction` AS __cf_direction,
    `cashflows.paymentCurrency_code` AS __cf_payment_currency,
    `cashflows.paymentAmount` AS __cf_payment_amount

LIMIT 3
```

Тогда одна исходная строка:

```text
id = 79524066
direction = ['PAY','RECEIVE']
paymentAmount = [10000000.000000,110547000.000000]
paymentCurrency = ['CNY','RUB']
```

становится двумя строками результата:

```text
Entity.id = 79524066:cashflows:1
direction = PAY
paymentAmount = 10000000.000000
paymentCurrency = CNY
```

```text
Entity.id = 79524066:cashflows:2
direction = RECEIVE
paymentAmount = 110547000.000000
paymentCurrency = RUB
```

Это намного лучше подходит для AG Grid, потому что каждая ячейка содержит scalar-значение.

## Можно ли вернуть массив как `['PAY','RECEIVE']`?

Да, можно. Например:

```sql
SELECT
    id AS `Entity.id`,
    portfolio_code AS `Trade.portfolio.code`,
    toJSONString(`cashflows.direction`) AS `Trade.cashflows.direction`,
    toJSONString(`cashflows.paymentCurrency_code`) AS `Trade.cashflows.paymentCurrency.code`,
    toJSONString(`cashflows.paymentAmount`) AS `Trade.cashflows.paymentAmount`
FROM `default`.`livebook322` FINAL
LIMIT 3
```

Результат будет примерно такой:

```json
{
  "Entity.id": 79524040,
  "Trade.portfolio.code": "FX_SB_STIR",
  "Trade.cashflows.direction": "[\"RECEIVE\",\"PAY\"]",
  "Trade.cashflows.paymentCurrency.code": "[\"CNY\",\"RUB\"]",
  "Trade.cashflows.paymentAmount": "[50,555075]"
}
```

Это позволит почти не менять логику формирования ответа.

Но важно: такой режим подходит только для просмотра данных как строка сделки.

То есть это нормально для такого сценария:

```json
{
  "columns": [
    { "fieldKey": "Trade.portfolio.code" },
    { "fieldKey": "Trade.cashflows.direction" }
  ],
  "rowGroupColumns": [],
  "valueColumns": [],
  "pivotColumns": []
}
```

Но это **не решает** такие сценарии:

```json
{
  "rowGroupColumns": [
    { "fieldKey": "Trade.cashflows.paymentDate" }
  ],
  "valueColumns": [
    { "fieldKey": "Trade.cashflows.paymentAmount", "aggFunc": "sum" }
  ],
  "pivotColumns": [
    { "fieldKey": "Trade.cashflows.direction" }
  ]
}
```

Потому что здесь нужно работать с отдельными элементами массива.

<details>
<summary><strong>Нужно ли делать отдельный класс для сложных таблиц?</strong></summary>

Можно, но я бы не рекомендовал делать полностью отдельный генератор с копированием всей логики.

Лучше сделать так:

```java
enum QueryMode {
    FLAT,
    ARRAYS_AS_JSON,
    ARRAY_JOIN
}
```

А затем внутри одного конвертера или через стратегии выбирать режим:

```java
interface V2SqlBuildStrategy {
    QueryMode mode();

    String build(V2ValidatedViewConfig validated);
}
```

Например:

```java
@Service
public class V2StreamViewConfigToSqlConverter {

    private final V2FlatSqlStrategy flatStrategy;
    private final V2ArraysAsJsonSqlStrategy arraysAsJsonStrategy;
    private final V2ArrayJoinSqlStrategy arrayJoinStrategy;

    public String convert(V2ValidatedViewConfig validated) {
        QueryMode mode = detectQueryMode(validated);

        return switch (mode) {
            case FLAT -> flatStrategy.build(validated);
            case ARRAYS_AS_JSON -> arraysAsJsonStrategy.build(validated);
            case ARRAY_JOIN -> arrayJoinStrategy.build(validated);
        };
    }
}
```

Но общие части лучше переиспользовать:

- построение фильтров;
- построение сортировок;
- построение `GROUP BY`;
- quoting identifiers;
- literals;
- aggregation expressions.

Иначе у вас быстро появятся два расходящихся генератора, которые будут по-разному обрабатывать фильтры, даты, сортировки и группировки.

</details>

<details>
<summary><strong>Нужен ли запрос к метаданным таблицы?</strong></summary>

Да, но не обязательно на каждый запрос.

Вам нужно знать для каждого поля:

- физическое имя колонки;
- тип поля;
- является ли колонка массивом;
- к какой repeated-группе относится массив, например `cashflows`;
- какой scalar-тип у элемента массива.

Например для ClickHouse можно один раз загрузить:

```sql
SELECT
    name,
    type
FROM system.columns
WHERE database = 'default'
  AND table = 'livebook322';
```

Для вашей таблицы результат будет примерно такой:

```text
name                                  type
------------------------------------  --------------------------------
id                                    Nullable(Int64)
globalId                              Int64
portfolio_code                        Nullable(String)
cashflows.direction                   Array(Nullable(String))
cashflows.paymentAmount               Array(Nullable(Decimal(18, 6)))
cashflows.paymentCurrency_code        Array(Nullable(String))
cashflows.paymentDate                 Array(Nullable(Date))
```

Дальше это нужно положить в ваши metadata/schema classes.

Например `V2SqlFieldRef` можно расширить:

```java
public record V2SqlFieldRef(
        String requestKey,
        String columnName,
        String alias,
        FieldKind kind,
        FieldMetadata metadata,

        boolean repeated,
        String repeatedGroup
) {
}
```

Для поля:

```text
Trade.cashflows.paymentAmount
```

будет:

```text
columnName = cashflows.paymentAmount
kind = BIG_DECIMAL
repeated = true
repeatedGroup = cashflows
```

Для поля:

```text
Trade.portfolio.code
```

будет:

```text
columnName = portfolio_code
kind = STRING
repeated = false
repeatedGroup = null
```

Важно: решение нужно принимать не по всей таблице, а по полям конкретного запроса.

Например таблица содержит массивы, но запрос такой:

```json
{
  "columns": [
    { "fieldKey": "Trade.portfolio.code" },
    { "fieldKey": "Entity.id" }
  ]
}
```

Тогда `ARRAY JOIN` не нужен.

</details>

<details>
<summary><strong>Как выбирать режим генерации SQL</strong></summary>

Примерная логика:

```java
private QueryMode detectQueryMode(V2ValidatedViewConfig validated) {
    JsonNode config = validated.config();

    Set<String> selectFields = collectFieldKeys(config, "columns");

    Set<String> relationalFields = new LinkedHashSet<>();
    relationalFields.addAll(collectFieldKeys(config, "rowGroupColumns"));
    relationalFields.addAll(collectFieldKeys(config, "pivotColumns"));
    relationalFields.addAll(collectFieldKeys(config, "valueColumns"));
    relationalFields.addAll(collectSortFields(config));
    relationalFields.addAll(collectFilterFields(config));
    relationalFields.addAll(collectGroupKeyFields(config));

    boolean selectedArrayField = selectFields.stream()
            .map(validated::field)
            .anyMatch(V2SqlFieldRef::repeated);

    boolean relationalArrayField = relationalFields.stream()
            .map(validated::field)
            .anyMatch(V2SqlFieldRef::repeated);

    if (relationalArrayField) {
        return QueryMode.ARRAY_JOIN;
    }

    if (selectedArrayField) {
        return QueryMode.ARRAYS_AS_JSON;
    }

    return QueryMode.FLAT;
}
```

То есть:

- если массивное поле участвует в `GROUP BY`, `pivot`, `aggregation`, `filter`, `sort` — нужен `ARRAY_JOIN`;
- если массивное поле только отображается в `columns` — можно вернуть его как JSON-строку;
- если массивных полей нет — обычный SQL.

</details>

## Важное ограничение: несколько разных массивных групп

Сейчас у вас есть одна группа:

```text
cashflows.*
```

Но если потом появится что-то вроде:

```text
fees.*
settlements.*
cashflows.*
```

то универсальный генератор должен понимать, что нельзя просто так делать:

```sql
ARRAY JOIN
    cashflows.amount,
    fees.amount
```

Это может либо дать ошибку, либо породить неправильную комбинацию данных.

Правильная логика:

- если в запросе массивные поля только из одной repeated-группы, например только `cashflows`, можно делать `ARRAY JOIN`;
- если из нескольких групп, например `cashflows` и `fees`, нужно либо:
  - запрещать такой запрос на уровне валидации;
  - либо явно понимать, какую сущность frontend хочет видеть;
  - либо строить несколько подзапросов;
  - либо делать отдельные flattened views.

## Хороший долгосрочный вариант: flat view для `cashflows`

Если frontend и текущая логика ответа сильно завязаны на плоские строки, можно создать ClickHouse view, которая разворачивает `cashflows`.

Например:

```sql
CREATE VIEW default.livebook322_cashflows_flat AS
SELECT
    concat(
        toString(ifNull(id, globalId)),
        ':cashflows:',
        toString(__cf_index)
    ) AS entity_id,

    id AS trade_id,
    globalId,
    portfolio_code,
    version,
    revision,
    savedAt,

    __cf_conversion_date AS cashflows_conversionDate,
    __cf_conversion_rate AS cashflows_conversionRate,
    __cf_conversion_source_alias AS cashflows_conversionSource_alias,
    __cf_counterparty_code AS cashflows_counterparty_code,
    __cf_direction AS cashflows_direction,
    __cf_notional_amount AS cashflows_notionalAmount,
    __cf_original_currency_code AS cashflows_originalCurrency_code,
    __cf_payment_amount AS cashflows_paymentAmount,
    __cf_payment_currency_code AS cashflows_paymentCurrency_code,
    __cf_payment_date AS cashflows_paymentDate,
    __cf_period_end AS cashflows_periodEnd,
    __cf_period_start AS cashflows_periodStart,
    __cf_state AS cashflows_state,
    __cf_type AS cashflows_type

FROM default.livebook322

ARRAY JOIN
    arrayEnumerate(`cashflows.paymentAmount`) AS __cf_index,
    `cashflows.conversionDate` AS __cf_conversion_date,
    `cashflows.conversionRate` AS __cf_conversion_rate,
    `cashflows.conversionSource_alias` AS __cf_conversion_source_alias,
    `cashflows.counterparty_code` AS __cf_counterparty_code,
    `cashflows.direction` AS __cf_direction,
    `cashflows.notionalAmount` AS __cf_notional_amount,
    `cashflows.originalCurrency_code` AS __cf_original_currency_code,
    `cashflows.paymentAmount` AS __cf_payment_amount,
    `cashflows.paymentCurrency_code` AS __cf_payment_currency_code,
    `cashflows.paymentDate` AS __cf_payment_date,
    `cashflows.periodEnd` AS __cf_period_end,
    `cashflows.periodStart` AS __cf_period_start,
    `cashflows.state` AS __cf_state,
    `cashflows.type` AS __cf_type;
```

Тогда ваш существующий генератор почти не нужно менять: для запросов по `Trade.cashflows.*` вы будете работать не с исходной таблицей `livebook322`, а с плоским view `livebook322_cashflows_flat`.

Минус: нужно правильно выбирать, какую physical source использовать:

- `livebook322` — если нужен уровень сделки;
- `livebook322_cashflows_flat` — если нужен уровень cashflow.

Плюс: SQL-генератор остаётся намного проще.

## Что я бы рекомендовал

Для вашей ситуации я бы сделал так:

1. **Добавить в метаданные поля признак `repeated` / `array`.**

2. **Добавить понятие repeated group**, например:

   ```text
   cashflows
   ```

3. **На этапе валидации определять режим запроса:**

   ```text
   FLAT
   ARRAYS_AS_JSON
   ARRAY_JOIN
   ```

4. **Для простого отображения массивов использовать `toJSONString(...)`.**

5. **Для группировок, pivot, фильтров и агрегаций по `cashflows.*` использовать `ARRAY JOIN`.**

6. **Не добавлять `id` / `globalId` в `GROUP BY` только ради уникальности строки.**  
   Для grouped rows нужно генерировать synthetic `Entity.id`.

7. **Для detail rows после `ARRAY JOIN` делать `Entity.id` из `id + array index`.**

Пример:

```sql
concat(toString(ifNull(id, globalId)), ':cashflows:', toString(__cf_index)) AS `Entity.id`
```

Для grouped rows:

```sql
concat(
    'g:',
    ifNull(toString(`portfolio_code`), 'NULL'),
    '|',
    ifNull(toString(__cf_direction), 'NULL'),
    '|',
    ifNull(toString(__cf_payment_currency), 'NULL')

```

## Итог

Да, универсальный генератор возможен, но он должен быть metadata-driven и mode-aware.

Самое важное:

- **таблица может содержать массивы, но это не значит, что всегда нужен `ARRAY JOIN`;**
- **если массивное поле участвует в агрегации, группировке, pivot или фильтре — нужен `ARRAY JOIN`;**
- **если массивное поле только отображается — можно вернуть его как JSON-строку через `toJSONString`;**
- **для AG Grid нельзя использовать обычный `id` после `ARRAY JOIN`, потому что он повторяется; нужен synthetic row id.**

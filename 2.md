Проблема **не в `internalArrayAlias(...)` и не в `ARRAY JOIN`**.

Эта ошибка возникает **до генерации SQL**, на этапе построения `V2LiveBookSchema`.

---

# Причина ошибки

Ошибка:

```json
"Cannot resolve LiveBook field 'Trade.authorizedBy.firstName' for liveBook 'LiveBook344'"
```

появляется вот здесь:

```java
private V2LiveBookSchema buildSchema(LiveBookDto liveBook, String tableName) {
    V2DataDictionaryPathResolver resolver =
            new V2DataDictionaryPathResolver(businessMetadataService.roots());

    Map<String, V2ResolvedLiveBookField> fieldsByPath = new LinkedHashMap<>();

    for (String fieldPath : liveBook.fields()) {
        try {
            V2ResolvedLiveBookField resolved = resolver.resolveScalar(fieldPath);
            fieldsByPath.put(fieldPath, resolved);
        } catch (Exception e) {
            throw new IllegalArgumentException(
                    "Cannot resolve LiveBook field '" + fieldPath +
                            "' for liveBook '" + liveBook.liveBookName() + "'",
                    e
            );
        }
    }

    return new V2LiveBookSchema(
            liveBook.liveBookId(),
            liveBook.liveBookName(),
            tableName,
            fieldsByPath
    );
}
```

То есть приложение пытается зарезолвить **все поля LiveBook**, а не только поля из текущего запроса.

В текущем запросе поля:

```json
"columns": [
  { "fieldKey": "Trade.portfolio.code" },
  { "fieldKey": "Cashflow.paymentDate" },
  { "fieldKey": "Cashflow.counterparty.code" },
  ...
]
```

Но где-то внутри `LiveBook344.fields()` есть поле:

```text
Trade.authorizedBy.firstName
```

И `V2DataDictionaryPathResolver` не может его обработать.

---

# Почему наличие колонки в ClickHouse не спасает

В таблице действительно есть колонка:

```sql
`authorizedBy_firstName` Nullable(String)
```

Но ошибка происходит раньше.

Сначала работает business resolver:

```java
resolver.resolveScalar("Trade.authorizedBy.firstName")
```

И только потом можно было бы связать это поле с физической колонкой ClickHouse.

То есть `system.columns` может показать, что колонка существует, но `V2DataDictionaryPathResolver` всё равно падает, потому что он не знает business path:

```text
Trade.authorizedBy.firstName
```

---

# Правильное исправление

У тебя есть два варианта.

## Вариант 1. Исправить business metadata

Это правильный долгосрочный вариант.

Нужно проверить, что в `BusinessMetadataService.roots()` существует путь:

```text
Trade.authorizedBy.firstName
```

Скорее всего, в metadata отсутствует что-то из этого:

```text
Trade.authorizedBy
authorizedBy.firstName
```

или поле называется иначе, например:

```text
Trade.authorizedUser.firstName
Trade.authorisedBy.firstName
Trade.authorizedBy.name
```

Если frontend / LiveBook ожидает путь:

```text
Trade.authorizedBy.firstName
```

то data dictionary должен уметь его резолвить.

---

## Вариант 2. Не падать из-за полей, которых нет в текущем запросе

Это, на мой взгляд, нужно сделать обязательно.

Сейчас один битый field в LiveBook ломает **любой запрос** к этому LiveBook, даже если поле вообще не используется.

Лучше резолвить только поля, которые реально есть в текущем config, либо пропускать unresolved поля, если они не запрошены.

---

# Минимальный безопасный патч

В `V2LiveBookSchemaService` измени `resolveSchema(...)` и `buildSchema(...)`.

## `resolveSchema(...)`

```java
public V2LiveBookSchema resolveSchema(JsonNode config) {
    LiveBookDto liveBook = resolveLiveBook(config);
    String tableName = tableNameResolver.resolve(config, liveBook.liveBookName());

    Set<String> requestedFieldKeys = collectRequestedFieldKeys(config);

    return buildSchema(liveBook, tableName, requestedFieldKeys);
}
```

---

## Новый `buildSchema(...)`

```java
private V2LiveBookSchema buildSchema(
        LiveBookDto liveBook,
        String tableName,
        Set<String> requestedFieldKeys
) {
    V2DataDictionaryPathResolver resolver =
            new V2DataDictionaryPathResolver(businessMetadataService.roots());

    Map<String, V2ResolvedLiveBookField> fieldsByPath = new LinkedHashMap<>();

    for (String fieldPath : liveBook.fields()) {
        String normalizedFieldPath = normalizeFieldKey(fieldPath);
        boolean requested = requestedFieldKeys.contains(normalizedFieldPath);

        try {
            V2ResolvedLiveBookField resolved = resolver.resolveScalar(fieldPath);
            fieldsByPath.put(fieldPath, resolved);
        } catch (Exception e) {
            if (requested) {
                throw new IllegalArgumentException(
                        "Cannot resolve requested LiveBook field '" + fieldPath +
                                "' for liveBook '" + liveBook.liveBookName() + "'",
                        e
                );
            }

            // Важно: поле есть в LiveBook, но не используется в текущем запросе.
            // Не надо из-за него ломать весь запрос.
            System.out.println(
                    "Skip unresolved non-requested LiveBook field '" +
                            fieldPath +
                            "' for liveBook '" +
                            liveBook.liveBookName() +
                            "': " +
                            e.getMessage()
            );
        }
    }

    return new V2LiveBookSchema(
            liveBook.liveBookId(),
            liveBook.liveBookName(),
            tableName,
            fieldsByPath
    );
}
```

Если ты уже добавил physical metadata в `V2LiveBookSchema`, тогда constructor будет с дополнительным аргументом — просто добавь его в `return`.

## Добавь `collectRequestedFieldKeys(...)`

```java
private Set<String> collectRequestedFieldKeys(JsonNode config) {
    Set<String> result = new LinkedHashSet<>();

    addFieldKeys(config.path("columns"), result);
    addFieldKeys(config.path("rowGroupColumns"), result);
    addFieldKeys(config.path("pivotColumns"), result);
    addFieldKeys(config.path("valueColumns"), result);
    addFieldKeys(config.path("groupKeys"), result);

    JsonNode sortModel = config.path("sortModel");

    if (sortModel != null && sortModel.isArray()) {
        for (JsonNode sortItem : sortModel) {
            String colId = sortItem.path("colId").asText(null);

            if (colId != null && !colId.isBlank()) {
                result.add(normalizeFieldKey(colId));
            }
        }
    }

    JsonNode filterModel = config.path("filterModel");

    if (filterModel != null && filterModel.isObject()) {
        filterModel.fieldNames().forEachRemaining(fieldKey ->
                result.add(normalizeFieldKey(fieldKey))
        );
    }

    return result;
}
```

---

## Добавь `addFieldKeys(...)`

```java
private void addFieldKeys(JsonNode arrayNode, Set<String> result) {
    if (arrayNode == null || !arrayNode.isArray()) {
        return;
    }

    for (JsonNode node : arrayNode) {
        String fieldKey = node.path("fieldKey").asText(null);

        if (fieldKey != null && !fieldKey.isBlank()) {
            result.add(normalizeFieldKey(fieldKey));
        }
    }
}
```

---

## Добавь `normalizeFieldKey(...)`

```java
private String normalizeFieldKey(String fieldKey) {
    return fieldKey == null
            ? ""
            : fieldKey.trim().toLowerCase(Locale.ROOT);
}
```

И не забудь импорт:

```java
import java.util.Set;
import java.util.LinkedHashSet;
import java.util.Locale;
```

# Ещё важный момент про `Entity.id`

У тебя в config приходит:

```json
{
  "fieldKey": "Entity.id"
}
```

Но в `V2ViewConfigValidator.resolveField(...)` сейчас специальная обработка есть только для:

```java
if ("id".equals(fieldKey)) {
    V2SqlFieldRef idRef = technicalIdRef();
    fields.put(fieldKey, idRef);
    return idRef;
}
```

Лучше исправить так:

```java
if (isTechnicalIdField(fieldKey)) {
    V2SqlFieldRef idRef = technicalIdRef();
    fields.put(fieldKey, idRef);
    return idRef;
}
```

И добавить метод:

```java
private boolean isTechnicalIdField(String fieldKey) {
    return "id".equalsIgnoreCase(fieldKey)
            || "entity.id".equalsIgnoreCase(fieldKey);
}
```

Иначе после исправления `Trade.authorizedBy.firstName` ты можешь получить новую ошибку уже по `Entity.id`, если такого поля нет в LiveBook metadata.

# Если реально входящий JSON содержит `"config"`

Ты показал JSON такого вида:

```json
{
  "success": false,
  "config": {
    "liveBookId": 344,
    ...
  },
  "type": "ConfigurationUpdated"
}
```

А твой `unwrapData(...)` сейчас умеет доставать только `"data"`:

```java
private JsonNode unwrapData(JsonNode requestOrConfig) {
    if (requestOrConfig != null
            && requestOrConfig.has("data")
            && requestOrConfig.get("data").isObject()) {
        return requestOrConfig.get("data");
    }

    return requestOrConfig;
}
```

Если такой wrapper реально приходит на вход validator-у, то нужно добавить поддержку `"config"`:

```java
private JsonNode unwrapData(JsonNode requestOrConfig) {
    if (requestOrConfig != null
            && requestOrConfig.has("data")
            && requestOrConfig.get("data").isObject()) {
        return requestOrConfig.get("data");
    }

    if (requestOrConfig != null
            && requestOrConfig.has("config")
            && requestOrConfig.get("config").isObject()) {
        return requestOrConfig.get("config");
    }

    return requestOrConfig;
}
```

Но судя по ошибке, `liveBookId` всё-таки был найден, поэтому, возможно, этот JSON — уже ответ клиенту, а не входной request.

# Итог

Текущая ошибка вызвана не SQL-генератором.

Причина здесь:

```java
for (String fieldPath : liveBook.fields()) {
    resolver.resolveScalar(fieldPath);
}
```

Ты резолвишь **все поля LiveBook344**, включая неиспользуемое в текущем запросе:

```text
Trade.authorizedBy.firstName
```

А resolver его не знает.

Что нужно сделать:

1. **Не валить весь запрос из-за unresolved поля, если оно не используется в текущем config.**
2. **Если поле используется — тогда честно возвращать validation error.**
3. **Отдельно исправить business metadata для `Trade.authorizedBy.firstName`, потому что физическая колонка `authorizedBy_firstName` в ClickHouse уже есть.**
4. **Добавить специальную обработку `Entity.id` как technical field.**

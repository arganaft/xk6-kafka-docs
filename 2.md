Твоя новая схема **в целом жизнеспособная**, особенно если:

- у каждого лайвбука своя схема;
- поля редко пересекаются;
- запросы частые;
- в будущем нужно физически разносить лайвбуки по разным машинам;
- можно договориться, чтобы в `topic A` всегда был указан `livebook_id`.

Но я бы внёс несколько важных поправок. Главные:

1. **Redis не должен быть источником истины.**  
   Redis лучше использовать как read-cache/read-model.

2. **Нужен один владелец состояния лайвбука.**  
   Иначе разные сервисы начнут независимо обновлять статус, и появятся гонки.

3. **Нужно явно зафиксировать границу между initial load и live updates.**  
   Иначе почти гарантированно появятся пропуски или дубли.

4. **Не стоит передавать большие инициализационные данные прямо через Kafka.**  
   Лучше передавать ссылку на staging/object storage/ClickHouse staging table.

5. **Сервис, который начинает писать данные для нового лайвбука в `topic A`, должен начинать с конкретного offset/watermark.**  
   Просто «после статуса готово начинаем писать» — опасно.

# 1. Как я понял твою новую идею

Ты предлагаешь примерно такой процесс:

```text
frontend
  ↓
gateway
  ↓
Kafka topic: LiveBook lifecycle events
  ↓
table creator service
  ↓
event: table created
  ↓
init data preparation service
  ↓
event: init data prepared
  ↓
init loader service
  ↓
event: init data loaded, table initialized
  ↓
producer service
  ↓
starts writing data for new livebook to topic A
  ↓
frontend receives status update
  ↓
Redis updated
```

Состояние лайвбука лежит в Redis:

```text
livebook_id
name
fields
types
status
created_at
schema_version
...
```

И многие сервисы читают Redis.

# 2. Что в этой схеме хорошо

## Хорошо, что есть событийная цепочка

То, что создание лайвбука происходит через Kafka-события, нормально:

```text
LiveBookCreationRequested
LiveBookTableCreated
LiveBookInitDataPrepared
LiveBookInitialized
LiveBookActivated
```

Это хорошо ложится на микросервисную архитектуру.

---

## Хорошо, что состояние транслируется на фронт

Фронт может видеть:

```text
создаём таблицу
готовим данные
загружаем данные
догружаем изменения
готово
```

Это правильно. Создание лайвбука — не мгновенная операция, поэтому асинхронный workflow здесь уместен.

---

## Хорошо, что можно договориться про `livebook_id` в `topic A`

Это сильно упрощает систему.

Если в `topic A` у каждого сообщения будет:

```json
{
  "livebook_id": "lb_123",
  "event_id": "event_456",
  "event_time": "2026-06-02T12:00:00Z",
  "payload": {}
}
```

то downstream-сервисам не нужно гадать, кому предназначены данные.

# 3. Главная проблема: Redis как источник истины

Я бы **не делал Redis главным хранилищем состояния лайвбука**.

Redis хорош как:

```text
быстрый cache
read model
projection
runtime config cache
```

Но плох как:

```text
источник истины для lifecycle-состояний
хранилище истории изменений
средство координации долгих процессов
```

Даже если Redis настроен с persistence, replication и backup, он всё равно менее удобен для таких задач, чем:

- PostgreSQL;
- отдельная metadata DB;
- compacted Kafka topic;
- event store;
- state store в Kafka Streams/Flink;
- комбинация `DB + outbox`.

---

## Как лучше

Я бы сделал так:

```text
Authoritative LiveBook Registry
```

Например, в PostgreSQL:

```sql
CREATE TABLE livebooks
(
    livebook_id text PRIMARY KEY,
    name text NOT NULL,
    status text NOT NULL,
    schema_version int NOT NULL,
    fields jsonb NOT NULL,
    created_at timestamptz NOT NULL,
    updated_at timestamptz NOT NULL,
    init_watermark jsonb,
    activation_watermark jsonb,
    error text
);
```

А Redis обновлял бы отдельный projector:

```text
livebook_registry_events
  ↓
redis projector
  ↓
Redis
```

То есть Redis — это кэш:

```text
source of truth → registry DB / compacted Kafka topic
cache/read model → Redis
```

# 4. Второй важный момент: кто имеет право менять статус?

Очень важно, чтобы статус лайвбука не меняли все сервисы напрямую.

Плохой вариант:

```text
table service пишет status = TABLE_CREATED
init service пишет status = INIT_READY
loader service пишет status = INITIALIZED
producer service пишет status = ACTIVE
redis updater тоже что-то пишет
```

Так можно получить гонки.

Например:

```text
loader ещё не загрузил данные,
но producer уже увидел старое/некорректное событие и начал писать live updates
```

Или:

```text
таблица создана частично,
но статус уже TABLE_CREATED
```

---

## Лучше: один координатор состояния

Я бы ввёл сервис:

```text
LiveBook Coordinator / LiveBook Orchestrator
```

Он один отвечает за state machine.

Остальные сервисы делают работу и отправляют результат:

```text
TableService → TableCreated
InitService → InitDataPrepared
LoaderService → InitialDataLoaded
ProducerControlService → ProducerEnabled
```

Но финальное решение о переходе статуса принимает координатор.

---

## Например

```text
Gateway
  ↓ command
LiveBook Coordinator
  ↓ event/command
Table Creator
  ↓ result
LiveBook Coordinator
  ↓ event/command
Init Preparer
  ↓ result
LiveBook Coordinator
  ↓ event/command
Init Loader
  ↓ result
LiveBook Coordinator
  ↓ event/command
Catch-up / Producer
  ↓ result
LiveBook Coordinator
  ↓
ACTIVE
```

То есть не так:

```text
сервис А написал событие,
сервис Б сам решил, что ему теперь делать,
сервис В сам обновил Redis
```

А так:

```text
сервис А сообщает результат,
координатор проверяет состояние,
координатор переводит лайвбук в следующий статус
```

# 5. Kafka: команды и события лучше разделить

Ты пишешь:

> gateway отправляет сообщение в топик кафка, на этот топик подписаны много сервисов, и каждый тип сообщения триггерит конкретный сервис

Это рабочий вариант, но со временем такой топик часто превращается в «общую шину всего».

Я бы разделил хотя бы концептуально:

## Commands

Команды — это намерение что-то сделать:

```text
CreateLiveBook
CreateLiveBookTable
PrepareLiveBookInitData
LoadLiveBookInitialData
ActivateLiveBook
```

Команда обычно имеет одного ответственного исполнителя.

---

## Events

События — это факт, который уже произошёл:

```text
LiveBookCreationRequested
LiveBookTableCreated
LiveBookInitDataPrepared
LiveBookInitialDataLoaded
LiveBookActivated
LiveBookCreationFailed
```

Событие может читать много потребителей.

---

## Пример

```text
livebook.commands
livebook.events
livebook.registry.compacted
```

Или хотя бы:

```text
livebook.lifecycle.events
```

с чёткой схемой сообщений.

# 6. Рекомендованная state machine лайвбука

Я бы сделал примерно такие состояния:

```text
REQUESTED
VALIDATING
TABLE_CREATING
TABLE_CREATED
INIT_PREPARING
INIT_READY
INIT_LOADING
INITIALIZED
CATCHING_UP
ACTIVE
FAILED
```

Можно упростить:

```text
REQUESTED
TABLE_CREATING
INITIALIZING
CATCHING_UP
ACTIVE
FAILED
```

Но я бы не убирал `CATCHING_UP`.

---

## Почему нужен `CATCHING_UP`

Потому что между:

```text
начали готовить initial data
```

и:

```text
лайвбук готов
```

могут прийти новые live-события.

Если их не обработать, будет дыра.

Например:

```text
12:00:00 начали подготовку initial data
12:00:10 пришло новое событие для livebook
12:01:00 initial data загрузились
12:01:05 лайвбук активировали
```

Вопрос:

```text
куда попало событие в 12:00:10?
```

Если никуда — данные потеряны.

# 7. Самая опасная часть твоей схемы

Вот эта часть:

> потом это читает другой сервис который записывает данные в топик А, и начинает кроме данных для других лайвбук записывать еще и данные для нового лайвбук

Это потенциально опасно.

Почему?

Потому что нужно понять:

```text
с какого момента начинать писать live-события для нового лайвбука?
```

Если просто «после того как таблица готова», то события, которые произошли во время инициализации, могут потеряться.

# 8. Как правильно связать initial load и live stream

Нужно ввести понятие:

```text
watermark
offset boundary
activation point
```

То есть при создании лайвбука вы должны зафиксировать границу:

```text
initial data покрывают данные до момента/offset X
live stream должен начинаться строго после X
```

---

## Вариант 1. Через Kafka offsets

Если source data идут из Kafka, лучше фиксировать offsets.

Например:

```json
{
  "livebook_id": "lb_123",
  "source_topic": "topic_A",
  "offsets": {
    "0": 12345,
    "1": 98765,
    "2": 55555
  }
}
```

Это означает:

```text
initial load покрывает данные до этих offsets включительно
live updates начинаются после этих offsets
```

---

## Вариант 2. Через event time

Можно использовать timestamp:

```json
{
  "livebook_id": "lb_123",
  "init_watermark_time": "2026-06-02T12:00:00Z"
}
```

Но timestamp хуже, потому что:

- события могут приходить с задержкой;
- event time может быть старым;
- порядок по времени не всегда совпадает с порядком обработки.

Kafka offsets обычно надёжнее.


После успешной загрузки:

```json
{
  "event_type": "LiveBookInitialDataLoaded",
  "livebook_id": "lb_123",
  "rows_count": 10500000,
  "checksum": "abc123",
  "loaded_until_offsets": {
    "topic_A": {
      "0": 12345,
      "1": 98765
    }
  }
}
```

---

## Шаг 7. Catch-up

Теперь нужно догрузить события, которые пришли после initial watermark.

Например:

```text
initial load покрывает topic_A до offsets:
partition 0 → 12345
partition 1 → 98765
```

А текущие offsets уже:

```text
partition 0 → 13000
partition 1 → 99500
```

Значит нужно догрузить:

```text
partition 0: 12346–13000
partition 1: 98766–99500
```

И только после этого лайвбук можно сделать `ACTIVE`.

---

## Шаг 8. Activation

Coordinator публикует:

```json
{
  "event_type": "LiveBookActivated",
  "livebook_id": "lb_123",
  "status": "ACTIVE",
  "schema_version": 1,
  "activated_at": "2026-06-02T12:05:00Z",
  "activation_offsets": {
    "topic_A": {
      "0": 13000,
      "1": 99500
    }
  }
}
```

После этого:

```text
frontend видит ACTIVE
Redis обновляется
producer пишет live data для этого livebook
service 2 пишет данные в таблицу
```

# 10. Ключевой вопрос: когда producer начинает писать данные для нового лайвбука?

Есть несколько вариантов.

---

## Вариант A. Producer начинает писать только после `ACTIVE`

Это похоже на твою идею.

Но тогда нужно гарантировать:

```text
initial data + catch-up покрывают всё до момента ACTIVE
```

То есть producer должен уметь начать не «с текущего момента», а с конкретного offset.

Например:

```text
LiveBookActivated содержит activation_offsets
producer начинает генерировать livebook events строго после этих offsets
```

Если producer не умеет replay по offset, этот вариант опасен.

---

## Вариант B. Producer начинает писать сразу после `REQUESTED`, но таблица ещё не активна

Это более надёжно, если есть промежуточный durable слой.

Например:

```text
producer пишет события для нового livebook в topic A/topic B сразу
consumer складывает их в raw/staging
после создания таблицы данные догружаются
после catch-up лайвбук становится ACTIVE
```

Плюс:

```text
нет потерь во время создания
```

Минус:

```text
нужно где-то хранить события до готовности таблицы
```

---

## Вариант C. Producer пишет после `TABLE_CREATED`, но до `ACTIVE`

Тоже можно.

Тогда сервис записи в ClickHouse должен понимать:

```text
если livebook не ACTIVE, писать в staging/raw
если ACTIVE, писать в основную таблицу
```

---

## Моя рекомендация

Если ты хочешь минимально менять текущую схему, делай так:

```text
producer начинает писать для нового livebook только после команды/события LiveBookActivated
```

Но обязательно:

```text
LiveBookActivated должен содержать activation_offsets/watermark
```

И producer должен уметь стартовать с этой границы.

Если producer не может стартовать с offset/watermark, тогда лучше начинать писать события раньше и буферизовать их в raw/staging.

# 11. Нужен ли временный топик?

В твоей новой схеме временный топик можно убрать, если есть одно из двух:

## Вариант 1. Есть replay из source topic

Если все события в `topic A` хранятся достаточно долго, и producer/consumer может переиграть события для конкретного лайвбука, временный топик не нужен.

---

## Вариант 2. Есть raw/staging storage

Например:

```text
topic A
  ↓
raw_events_by_livebook
  ↓
per-livebook table
```

Тогда если таблица ещё не готова, данные не теряются.

# 12. Redis в твоей схеме

Использовать Redis можно, но я бы сделал так:

```text
LiveBook Coordinator
  ↓
livebook.registry.events
  ↓
Redis Projector
  ↓
Redis
```

А не так:

```text
каждый сервис сам обновляет Redis
```

---

## Почему

Если каждый сервис пишет в Redis, появятся проблемы:

```text
service A записал TABLE_CREATED
service B записал INIT_LOADING
service C из-за retry снова записал TABLE_CREATED
frontend увидел откат состояния
```

Поэтому лучше:

```text
один поток событий состояния
один projector в Redis
```

---

## В Redis можно хранить

```json
{
  "livebook_id": "lb_123",
  "name": "livebook1",
  "status": "ACTIVE",
  "schema_version": 1,
  "table_name": "livebook_lb_123",
  "fields": [
    {
      "name": "price",
      "type": "Float64"
    },
    {
      "name": "status",
      "type": "String"
    }
  ],
  "created_at": "2026-06-02T12:00:00Z",
  "updated_at": "2026-06-02T12:05:00Z"
}
```

Но source of truth лучше держать в более надёжном хранилище.

# 13. Отдельная таблица на лайвбук — нормально в твоём случае

С учётом твоих вводных отдельные таблицы выглядят оправданно:

- разные схемы;
- разный набор полей;
- разные источники данных;
- частые запросы;
- потенциальное разнесение по машинам;
- низкое пересечение колонок.

То есть единая широкая таблица вида:

```text
livebook_id, field_1, field_2, ..., field_10000
```

тут может быть неудобной.

---

## Но есть нюансы ClickHouse

Если таблиц будет много, нужно заранее подумать про:

1. **Количество таблиц.**  
   Сотни — нормально. Тысячи — уже надо аккуратно. Десятки тысяч — может стать больно.

2. **DDL на кластере.**  
   `CREATE TABLE ON CLUSTER` может быть не мгновенным.

3. **Миграции схем.**  
   У каждого лайвбука может быть своя версия схемы.

4. **Мониторинг.**  
   Нужно понимать, какая таблица где находится, сколько в ней строк, какой lag, какие ошибки.

5. **Удаление лайвбуков.**  
   Нужен lifecycle:

   ```text
   ACTIVE → ARCHIVING → ARCHIVED → DELETING → DELETED
   ```

6. **Распределение по машинам.**  
   Нужен registry, где написано:

   ```text
   livebook_id → cluster/shard/database/table
   ```

# 14. Как хранить схему лайвбука

Так как у каждого лайвбука свой набор полей, я бы обязательно ввёл:

```text
schema_version
schema_hash
field definitions
```

Пример:

```json
{
  "livebook_id": "lb_123",
  "schema_version": 3,
  "schema_hash": "sha256:abc123",
  "fields": [
    {
      "name": "price",
      "type": "Float64",
      "nullable": true,
      "source": "kafka"
    },
    {
      "name": "user_segment",
      "type": "LowCardinality(String)",
      "nullable": true,
      "source": "crm"
    },
    {
      "name": "score",
      "type": "Float32",
      "nullable": false,
      "source": "ml_service"
    }
  ]
}
```

Важно, чтобы каждое событие с данными содержало:

```text
livebook_id
schema_version
event_id
```

Иначе при изменении схемы будет сложно понять, как интерпретировать событие.

# 15. Данные из Kafka и из других источников

Ты сказал важную вещь:

> у лайвбука есть поля, которые приходят не из топика Kafka, а из других источников

Тогда инициализация лайвбука — это не просто загрузка из Kafka. Это сборка состояния из нескольких источников.

Например:

```text
Kafka topic A
CRM
PostgreSQL
S3
ML service
external API
```

Для таких случаев я бы делал initial load через staging.

---

## Пример

```text
Init Preparer
  ↓
читает Kafka
читает CRM
читает PostgreSQL
читает ML service
  ↓
собирает staging dataset
  ↓
пишет в ClickHouse staging table или S3
  ↓
публикует InitDataPrepared
```

То есть:

```text
Kafka event сообщает не сами данные, а где лежит подготовленный snapshot
```

# 16. Как избежать дублей

Все события должны иметь стабильный `event_id`.

Например:

```text
event_id = source_name + source_entity_id + livebook_id + schema_version + event_version
```

Или для Kafka:

```text
event_id = source_topic + source_partition + source_offset + livebook_id
```

В таблице ClickHouse лучше хранить:

```sql
event_id String,
source_topic String,
source_partition UInt32,
source_offset UInt64,
schema_version UInt32,
ingest_time DateTime64(3)
```

Для идемпотентности можно использовать:

```sql
ENGINE = ReplacingMergeTree(version)
ORDER BY (event_id)
```

или:

```sql
ENGINE = ReplacingMergeTree(version)
ORDER BY (business_key, event_time, event_id)
```

Конкретный `ORDER BY` зависит от запросов.

---

# 17. Как избежать пропусков

Минимальный набор:

```text
1. initial load имеет source_offsets/watermark
2. live stream стартует после этих offsets
3. catch-up догружает всё между initial watermark и activation watermark
4. status ACTIVE выставляется только после catch-up
```

Без этого консистентности не будет.

# 19. Нужно ли gateway публиковать событие напрямую?

Я бы аккуратно разделил:

## Нежелательно

```text
gateway → LiveBookCreationRequested event
```

Потому что событие — это факт. Gateway не всегда должен быть владельцем бизнес-факта.

---

## Лучше

```text
gateway → CreateLiveBook command
coordinator → LiveBookCreationRequested event
```

То есть gateway говорит:

```text
пользователь хочет создать лайвбук
```

А coordinator говорит:

```text
запрос принят, лайвбук зарегистрирован
```

# 20. Один общий lifecycle topic или несколько?

Можно начать с одного топика:

```text
livebook.lifecycle.events
```

Но важно:

- ключ сообщения должен быть `livebook_id`;
- все события одного лайвбука должны попадать в одну partition;
- все consumer должны быть идемпотентными;
- должна быть схема событий;
- желательно использовать schema registry.

Ключ:

```text
key = livebook_id
```

Это даст порядок событий внутри одного лайвбука.

---

## Пример порядка

```text
key = lb_123

LiveBookCreationRequested
LiveBookTableCreated
LiveBookInitDataPrepared
LiveBookInitialDataLoaded
LiveBookActivated
```

Kafka сохранит порядок этих событий для `lb_123`.

# 21. Что делать при ошибках

Для каждого этапа должен быть failure event:

```text

LiveBookInitPreparationFailed
LiveBookInitialLoadFailed
LiveBookActivationFailed
```

Пример:

```json

  "event_type": "LiveBookInitialLoadFailed",
  "livebook_id": "lb_123",

  "error_code": "CHECKSUM_MISMATCH",

```

```text

```

или:

```text

```

# 22. Idempotency обязательна

Kafka почти всегда означает:

```text
at-least-once processing
```

То есть сервис может обработать одно событие два раза.

Поэтому:

## Table creator

Должен уметь безопасно повторить:

```sql
CREATE TABLE IF NOT EXISTS ...
```

---

## Init loader

Должен не загрузить данные дважды.

Варианты:

1. грузить сначала в staging;
2. проверять `init_batch_id`;
3. использовать `event_id`;
4. использовать `ReplacingMergeTree`;
5. хранить таблицу обработанных команд.

---

## Coordinator

Должен проверять переходы состояния.

Например:

```text
если статус уже ACTIVE,
повторное LiveBookInitialDataLoaded игнорируется
```

# 23. Нужны version/CAS-переходы состояния

В registry полезно хранить версию:

```text
state_version
```

И делать переходы так:

```sql
UPDATE livebooks
SET status = 'INIT_LOADING',
    state_version = state_version + 1
WHERE livebook_id = 'lb_123'
  AND status = 'INIT_READY'
  AND state_version = 7;
```

Если обновилось $0$ строк — значит состояние уже изменилось другим процессом.

Это защищает от гонок.

# 26. Если коротко про твою схему

Твоя схема хороша как направление, но я бы изменил её так:

| В твоей схеме | Как я бы сделал |
|---|---|
| Redis хранит состояние | Redis — только cache/read-model |
| Gateway пишет lifecycle event | Gateway пишет command, coordinator пишет event |
| Каждый сервис триггерит следующий | Лучше coordinator/saga управляет переходами |
| Данные инициализации в Kafka | В Kafka только ссылка на staging/object storage |
| Producer начинает писать после готовности | Только с явным `activation_offsets` |
| Redis обновляется по ходу процесса | Redis обновляет отдельный projector |
| Статусы меняют разные сервисы | Статус меняет один owner — coordinator |

# 27. Самое важное правило

Главный риск в такой системе — не создание таблицы. Главный риск — вот эта граница:

```text
initial data закончились здесь
live updates начались здесь
```

Если эта граница не формализована, будут:

```text
пропуски
дубли
неконсистентные таблицы
сложные ручные восстановления
```

Поэтому в проектировании я бы первым делом зафиксировал:

```text
какой offset/watermark покрывает initial load
с какого offset/watermark начинается live stream
как выполнить catch-up
когда можно поставить ACTIVE
```

# 28. Моя финальная рекомендация

Я бы строил так:

```text
1. Данные в topic A всегда содержат livebook_id.

2. Создание лайвбука идёт через LiveBook Coordinator.

3. Redis используется только как кэш состояния.

4. Источник истины — registry DB или compacted Kafka topic.

5. Таблица создаётся отдельным Table Creator Service.

6. Initial data готовятся в staging, а в Kafka передаётся ссылка.

7. Initial load обязательно содержит source_offsets/watermark.

8. После initial load выполняется catch-up.

9. Только после catch-up лайвбук получает статус ACTIVE.

10. Только после ACTIVE frontend видит лайвбук как готовый, а producer/writer работает с ним как с полноценным.
```

Если это соблюсти, твоя архитектура будет достаточно надёжной и масштабируемой.

Да, проблема **именно в `WebSocketConfig`**, в этом месте:

```java
return MessageBuilder.createMessage(
        obje,
        message.getHeaders()
);
```

Ты подменяешь payload у служебного Spring-сообщения `CONNECT_ACK`. В результате клиент получает STOMP-frame `CONNECTED` **с body**, а Spring STOMP client строго валидирует протокол и падает:

```text
StompConversionException: CONNECTED shouldn't have a payload
```

`CONNECTED` frame в STOMP не должен иметь payload. Session id нужно отдавать **в headers**, либо отдельным `MESSAGE` после подключения, но не body внутри `CONNECTED`.

## 1. Исправь `WebSocketConfig`

Текущий код фактически отправляет JSON-body в `CONNECTED`. Нужно оставить payload исходным и добавить session id как STOMP native header.

Например так:

```java
package com.pivotService.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.messaging.Message;
import org.springframework.messaging.MessageChannel;
import org.springframework.messaging.simp.SimpMessageType;
import org.springframework.messaging.simp.config.ChannelRegistration;
import org.springframework.messaging.simp.config.MessageBrokerRegistry;
import org.springframework.messaging.simp.stomp.StompHeaderAccessor;
import org.springframework.messaging.support.ChannelInterceptor;
import org.springframework.messaging.support.MessageBuilder;
import org.springframework.scheduling.concurrent.ThreadPoolTaskScheduler;
import org.springframework.web.socket.config.annotation.EnableWebSocket;
import org.springframework.web.socket.config.annotation.EnableWebSocketMessageBroker;
import org.springframework.web.socket.config.annotation.StompEndpointRegistry;
import org.springframework.web.socket.config.annotation.WebSocketMessageBrokerConfigurer;

@Configuration
@EnableWebSocketMessageBroker
@EnableWebSocket
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    public static final String CONNECTED_SESSION_ID_HEADER = "x-simp-session-id";

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("*");
    }

    @Override
    public void configureClientOutboundChannel(ChannelRegistration registration) {
        registration.interceptors(new ChannelInterceptor() {

            @Override
            public Message<?> preSend(Message<?> message, MessageChannel channel) {
                StompHeaderAccessor accessor = StompHeaderAccessor.wrap(message);

                if (!SimpMessageType.CONNECT_ACK.equals(accessor.getMessageType())) {
                    return message;
                }

                String sessionId = accessor.getSessionId();

                if (sessionId != null) {
                    accessor.setNativeHeader(CONNECTED_SESSION_ID_HEADER, sessionId);
                }

                /*
                 * Важно:
                 * payload оставляем исходным.
                 * Нельзя класть JSON body в CONNECTED frame.
                 */
                return MessageBuilder.createMessage(
                        message.getPayload(),
                        accessor.getMessageHeaders()
                );
            }
        });
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        ThreadPoolTaskScheduler taskScheduler = new ThreadPoolTaskScheduler();
        taskScheduler.setPoolSize(1);
        taskScheduler.setThreadNamePrefix("heartbeat-");
        taskScheduler.initialize();

        registry.enableSimpleBroker("/topic", "/queue")
                .setTaskScheduler(taskScheduler)
                .setHeartbeatValue(new long[]{10000, 10000});

        registry.setApplicationDestinationPrefixes("/app");
        registry.setUserDestinationPrefix("/user");
    }
}
```

После этого frontend сможет читать session id из headers `CONNECTED` frame.

Например в STOMP.js:

```javascript
client.onConnect = (frame) => {
  const sessionId = frame.headers['x-simp-session-id'];
  console.log('Spring websocket session id:', sessionId);
};
```

Твой старый код делал вот это:

Тест даже не доходит до отправки твоего JSON-запроса.

## 3. Можно добавить проверку session id в тест

Если хочешь дополнительно проверить, что session id действительно пришёл в `CONNECTED` headers, добавь очередь для headers.

### Поле в тесте

```java
private BlockingQueue<StompHeaders> connectedFrames;
```

### В `connectToWebSocket()`

```java
@BeforeEach
void connectToWebSocket() throws Exception {
    contractObjectMapper = applicationObjectMapper.copy()
            .enable(DeserializationFeature.USE_BIG_DECIMAL_FOR_FLOATS);

    contractObjectMapper.setNodeFactory(JsonNodeFactory.withExactBigDecimals(true));

    receivedFrames = new LinkedBlockingQueue<>();
    connectedFrames = new LinkedBlockingQueue<>();
    asyncErrors = new CopyOnWriteArrayList<>();

    MappingJackson2MessageConverter messageConverter = new MappingJackson2MessageConverter();
    messageConverter.setObjectMapper(contractObjectMapper);

    stompClient = new WebSocketStompClient(new StandardWebSocketClient());
    stompClient.setMessageConverter(messageConverter);
    stompClient.setInboundMessageSizeLimit(2 * 1024 * 1024);

    stompSession = stompClient
            .connectAsync(
                    "ws://localhost:%d/ws".formatted(port),
                    new RecordingStompSessionHandler()
            )
            .get(3, SECONDS);

    StompHeaders connectedHeaders = connectedFrames.poll(1, SECONDS);

    then(connectedHeaders)
            .as("CONNECTED frame headers")
            .isNotNull();

    then(connectedHeaders.getFirst("x-simp-session-id"))
            .as("Spring simp session id from CONNECTED frame")
            .isNotBlank();

    stompSession.subscribe(DASHBOARD_MESSAGES_DESTINATION, new JsonFrameHandler());
}
```

### Обнови `RecordingStompSessionHandler`

```java
private final class RecordingStompSessionHandler extends StompSessionHandlerAdapter {

    @Override
    public void afterConnected(StompSession session, StompHeaders connectedHeaders) {
        TradeDashboardWebSocketContractTest.this.connectedFrames.add(connectedHeaders);
    }

    @Override
    public void handleException(
            StompSession session,
            StompCommand command,
            StompHeaders headers,
            byte[] payload,
            Throwable exception
    ) {
        asyncErrors.add(exception);
    }

    @Override
    public void handleTransportError(StompSession session, Throwable exception) {
        asyncErrors.add(exception);
    }
}
```

## 5. Важное замечание про `executeSubscriptionQuery.executeSubscriptionQuery()`

Сейчас у тебя вызов стоит после ожидания initial data:

```java
sendJson(CONFIG_REQUEST_DESTINATION, "set-view-configuration.json");

JsonNode configurationUpdated = nextFrame("ConfigurationUpdated");
JsonNode initialData = nextFrame("initial ReceiveData");

assertConfigurationUpdated(configurationUpdated);
assertReceiveData(initialData, "receive-data.initial.expected.json");

executeSubscriptionQuery.executeSubscriptionQuery();
```

Если initial data отправляется только после вызова `executeSubscriptionQuery.executeSubscriptionQuery()`, то порядок должен быть другим:

```java
sendJson(CONFIG_REQUEST_DESTINATION, "set-view-configuration.json");

JsonNode configurationUpdated = nextFrame("ConfigurationUpdated");

executeSubscriptionQuery.executeSubscriptionQuery();

JsonNode initialData = nextFrame("initial ReceiveData");

assertConfigurationUpdated(configurationUpdated);
assertReceiveData(initialData, "receive-data.initial.expected.json");
```

Если же backend действительно сам сразу после `SetViewConfiguration` отправляет initial data, тогда `executeSubscriptionQuery.executeSubscriptionQuery()` здесь вообще не нужен для этого теста.

То есть выбери один из вариантов:

### Вариант A — initial data отправляется автоматически

```java
sendJson(CONFIG_REQUEST_DESTINATION, "set-view-configuration.json");

JsonNode configurationUpdated = nextFrame("ConfigurationUpdated");
JsonNode initialData = nextFrame("initial ReceiveData");

assertConfigurationUpdated(configurationUpdated);
assertReceiveData(initialData, "receive-data.initial.expected.json");
```

### Вариант B — initial/update data отправляется вручную через `ExecuteSubscriptionQuery`

```java
sendJson(CONFIG_REQUEST_DESTINATION, "set-view-configuration.json");

JsonNode configurationUpdated = nextFrame("ConfigurationUpdated");

executeSubscriptionQuery.executeSubscriptionQuery();

JsonNode initialData = nextFrame("initial ReceiveData");

assertConfigurationUpdated(configurationUpdated);
assertReceiveData(initialData, "receive-data.initial.expected.json");
```

## 7. Резюме

Главная причина ошибки:

```java
CONNECTED shouldn't have a payload
```

— backend отправляет JSON-body внутри STOMP `CONNECTED` frame.

Правильное исправление:

- не менять payload у `CONNECT_ACK`;
- session id положить в STOMP header;
- для JSON-событий использовать обычные `MESSAGE` frames, а не `CONNECTED`.

После этого `WebSocketStompClient` сможет подключиться, и тест дойдёт до проверки `ConfigurationUpdated` и `ReceiveData`.

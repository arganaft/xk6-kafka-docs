Да, это поведение очень показательное.

## Что произошло

### Вариант 1 — с `configureClientOutboundChannel()`

Ошибка:

```text
ConnectionLostException: Connection closed
```

на строке:

```java
.connectAsync(...).get(...)
```

означает: **STOMP CONNECT не завершился успешно**. Тест даже не дошёл до подписки и бизнес-логики.

То есть проблема всё ещё в кастомизации служебного `CONNECT_ACK` / `CONNECTED` frame.

Даже если ты больше не кладёшь JSON payload в `CONNECTED`, сама попытка модифицировать служебное сообщение через:

```java
StompHeaderAccessor accessor = StompHeaderAccessor.wrap(message);
accessor.setNativeHeader(CONNECTED_SESSION_ID_HEADER, sessionId);

return MessageBuilder.createMessage(
        message.getPayload(),
        accessor.getMessageHeaders()
);
```

в твоей связке Spring Boot `3.5.5` / Spring `6.2.10` / Jetty приводит к некорректному завершению соединения.

**Практический вывод:** для этого контрактного теста и вообще для бизнес-логики дашборда лучше убрать этот interceptor.

---

### Вариант 2 — без `configureClientOutboundChannel()`

Ошибка стала другой:

```text
java.lang.IllegalArgumentException: To track receipts, a TaskScheduler must be configured
```

Это уже **не ошибка backend WebSocketConfig**.

Это ошибка настройки **тестового STOMP-клиента**.

Она возникает из-за этого кода:

```java
stompSession.setAutoReceipt(true);
```

и/или из-за:

```java
subscription.addReceiptTask(...)
```

Если STOMP-клиент должен отслеживать `RECEIPT` frames, ему нужен `TaskScheduler`.

То есть второй вариант — это прогресс: **соединение уже успешно установилось**, но тестовый клиент неправильно настроен для receipt tracking.

# 1. Рекомендуемый `WebSocketConfig`

Я бы оставил production config пока в таком виде:

```java
package com.pivotService.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.messaging.simp.config.MessageBrokerRegistry;
import org.springframework.scheduling.concurrent.ThreadPoolTaskScheduler;
import org.springframework.web.socket.config.annotation.EnableWebSocket;
import org.springframework.web.socket.config.annotation.EnableWebSocketMessageBroker;
import org.springframework.web.socket.config.annotation.StompEndpointRegistry;
import org.springframework.web.socket.config.annotation.WebSocketMessageBrokerConfigurer;

@Configuration
@EnableWebSocketMessageBroker
@EnableWebSocket
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("*");
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        ThreadPoolTaskScheduler taskScheduler = new ThreadPoolTaskScheduler();
        taskScheduler.setPoolSize(1);
        taskScheduler.setThreadNamePrefix("heartbeat-");
        taskScheduler.initialize();

        registry.enableSimpleBroker("/topic", "/queue")
                .setTaskScheduler(taskScheduler)
                .setHeartbeatValue(new long[]{10000, 10000});

        registry.setApplicationDestinationPrefixes("/app");
        registry.setUserDestinationPrefix("/user");
    }
}
```

Для твоего кейса frontend не обязан знать Spring `simpSessionId`.

Backend и так получает session id здесь:

```java
@MessageMapping("/v2/config")
public void handleClientRequestV2(
        @Payload JsonNode request,
        SimpMessageHeaderAccessor headerAccessor
) throws SQLException {
    String sessionId = headerAccessor.getSessionId();
    v2RequestProcessDispatcher.dispatchClientRequest(request, headerAccessor);
}
```

И дальше этот session id можно сохранить в `SessionStorage`.

# 2. Исправь тестовый STOMP-клиент

Добавь scheduler для receipts.

## Поле в тестовом классе

```java
private ThreadPoolTaskScheduler stompClientTaskScheduler;
```

Импорт:

```java
import org.springframework.scheduling.concurrent.ThreadPoolTaskScheduler;
```

---

## Исправленный `connectToWebSocket()`

```java
@BeforeEach
void connectToWebSocket() throws Exception {
    contractObjectMapper = applicationObjectMapper.copy()
            .enable(DeserializationFeature.USE_BIG_DECIMAL_FOR_FLOATS);

    contractObjectMapper.setNodeFactory(JsonNodeFactory.withExactBigDecimals(true));

    receivedFrames = new LinkedBlockingQueue<>();
    asyncErrors = new CopyOnWriteArrayList<>();

    MappingJackson2MessageConverter messageConverter = new MappingJackson2MessageConverter();
    messageConverter.setObjectMapper(contractObjectMapper);

    stompClientTaskScheduler = new ThreadPoolTaskScheduler();
    stompClientTaskScheduler.setPoolSize(1);
    stompClientTaskScheduler.setThreadNamePrefix("test-stomp-receipt-");
    stompClientTaskScheduler.initialize();

    stompClient = new WebSocketStompClient(new StandardWebSocketClient());
    stompClient.setMessageConverter(messageConverter);
    stompClient.setTaskScheduler(stompClientTaskScheduler);

    /*
     * Heartbeat в этом тесте нам не нужен.
     * Scheduler оставляем именно для receipt tracking.
     */
    stompClient.setDefaultHeartbeat(new long[]{0, 0});

    stompClient.setInboundMessageSizeLimit(2 * 1024 * 1024);

    stompSession = stompClient
            .connectAsync(
                    "ws://localhost:%d/ws".formatted(port),
                    new RecordingStompSessionHandler()
            )
            .get(5, SECONDS);

    stompSession.setAutoReceipt(true);

    CountDownLatch subscribed = new CountDownLatch(1);

    StompSession.Subscription subscription = stompSession.subscribe(
            DASHBOARD_MESSAGES_DESTINATION,
            new JsonFrameHandler()
    );

    subscription.addReceiptTask(subscribed::countDown);

    subscription.addReceiptLostTask(() ->
            asyncErrors.add(new AssertionError(
                    "Subscription receipt was lost for destination: " + DASHBOARD_MESSAGES_DESTINATION
            ))
    );

    then(subscribed.await(5, SECONDS))
            .as("Subscription to %s must be registered before sending request", DASHBOARD_MESSAGES_DESTINATION)
            .isTrue();
}
```

Нужный импорт:

```java
import java.util.concurrent.CountDownLatch;
```

---

## Исправленный `disconnectFromWebSocket()`

```java
@AfterEach
void disconnectFromWebSocket() {
    if (stompSession != null && stompSession.isConnected()) {
        stompSession.disconnect();
    }

    if (stompClient != null) {
        stompClient.stop();
    }

    if (stompClientTaskScheduler != null) {
        stompClientTaskScheduler.shutdown();
    }
}
```

# 3. Убери проверку `x-simp-session-id` из теста

Раз ты убираешь interceptor из `WebSocketConfig`, то это больше не нужно:

```java
private BlockingQueue<StompHeaders> connectedFrames;
```

И это тоже больше не нужно:

```java
StompHeaders connectedHeaders = connectedFrames.poll(1, SECONDS);

then(connectedHeaders)
        .as("CONNECTED frame headers")
        .isNotNull();

then(connectedHeaders.getFirst("x-simp-session-id"))
        .as("Spring simp session id from CONNECTED frame")
        .isNotBlank();
```

И из `RecordingStompSessionHandler` убери:

```java
@Override
public void afterConnected(StompSession session, StompHeaders connectedHeaders) {
    TradeDashboardWebSocketContractTest.this.connectedFrames.add(connectedHeaders);
}
```

# 4. Итоговый смысл ошибки

## Было

```text
CONNECTED shouldn't have a payload
```

Причина: backend отправлял JSON body внутри STOMP `CONNECTED`.

## Потом стало

```text
ConnectionLostException: Connection closed
```

Причина: backend всё ещё нестабильно модифицирует служебный `CONNECT_ACK`.

## Без interceptor-а стало

```text
To track receipts, a TaskScheduler must be configured
```

Причина: STOMP connect уже прошёл, но тестовый клиент попросил receipts без scheduler-а.

Это значит, что правильное направление такое:

1. оставить `WebSocketConfig` без outbound interceptor-а;
2. настроить `TaskScheduler` в тестовом `WebSocketStompClient`;
3. продолжить тестировать уже бизнес-контракт `ConfigurationUpdated` и `ReceiveData`.

# 5. Если frontend всё-таки нужен session id

Лучше не передавать его через `CONNECTED`.

Сделай отдельное обычное STOMP-сообщение.

Например frontend:

```javascript
client.subscribe('/user/queue/session', message => {
  console.log(JSON.parse(message.body));
});

client.publish({
  destination: '/app/session',
  body: '{}'
});
```

Backend:

```java
package com.pivotService.controller;

import com.fasterxml.jackson.core.JsonProcessingException;
import com.fasterxml.jackson.databind.ObjectMapper;
import com.pivotService.config.model.ConnectedEvent;
import com.pivotService.controller.sender.MessageSender;
import lombok.RequiredArgsConstructor;
import org.springframework.messaging.handler.annotation.MessageMapping;
import org.springframework.messaging.simp.SimpMessageHeaderAccessor;
import org.springframework.stereotype.Controller;

@Controller
@RequiredArgsConstructor
public class WebSocketSessionController {

    private final ObjectMapper objectMapper;
    private final MessageSender messageSender;

    @MessageMapping("/session")
    public void sendSessionInfo(SimpMessageHeaderAccessor headerAccessor) throws JsonProcessingException {
        String sessionId = headerAccessor.getSessionId();

        ConnectedEvent event = new ConnectedEvent(sessionId);

        messageSender.sendToSession(
                sessionId,
                objectMapper.writeValueAsString(event)
        );
    }
}
```

Но для твоего текущего дашборд-кейса это, скорее всего, вообще не нужно.

# 6. Что делать сейчас

Минимальный план:

1. Оставь `WebSocketConfig` **без** `configureClientOutboundChannel()`.
2. Добавь `ThreadPoolTaskScheduler` в тестовый `WebSocketStompClient`.
3. Убери проверку `x-simp-session-id`.
4. Запусти тест снова.

Следующая возможная ошибка, если она появится, скорее всего будет уже не про connect, а про одно из двух:

- тест подписан не на тот destination;
- backend отправляет payload не в том формате / не туда.

Но текущая ошибка с receipts решается именно настройкой `TaskScheduler` на тестовом клиенте.

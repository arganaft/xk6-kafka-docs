## 1. Самая вероятная причина

Почти наверняка сервер закрывает соединение во время обработки `CONNECT_ACK`.

Раньше была явная ошибка:

```text
CONNECTED shouldn't have a payload
```

После исправлений ошибка стала другой:

```text
Connection closed
```

Это значит: payload в `CONNECTED` ты, скорее всего, уже убрал, но в interceptor-е `configureClientOutboundChannel()` всё ещё происходит что-то, из-за чего сервер не может нормально отправить `CONNECTED` frame.

Особенно подозрителен вот этот стек после закрытия:

```text
IndexOutOfBoundsException: Index 1 out of bounds for length 1
at org.springframework.messaging.support.AbstractMessageChannel$ChannelInterceptorChain.triggerAfterSendCompletion
```

Такое часто появляется как побочный эффект ошибки в `ChannelInterceptor` на outbound channel.

## 3. Но главное — исправь `WebSocketConfig`

Для твоего бизнес-кейса **вообще не нужно отправлять session id в `CONNECTED` frame**.

Spring и так передаёт session id в контроллер через:

```java
SimpMessageHeaderAccessor headerAccessor
```

Например:

```java
String sessionId = headerAccessor.getSessionId();
```

То есть backend может сохранить подписку на конкретную websocket session без того, чтобы frontend знал этот session id.

### Лучший вариант сейчас — временно полностью убрать outbound interceptor

Сделай `WebSocketConfig` вот таким:

```java
package com.pivotService.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.messaging.simp.config.MessageBrokerRegistry;
import org.springframework.scheduling.concurrent.ThreadPoolTaskScheduler;
import org.springframework.web.socket.config.annotation.EnableWebSocket;
import org.springframework.web.socket.config.annotation.EnableWebSocketMessageBroker;
import org.springframework.web.socket.config.annotation.StompEndpointRegistry;
import org.springframework.web.socket.config.annotation.WebSocketMessageBrokerConfigurer;

@Configuration
@EnableWebSocketMessageBroker
@EnableWebSocket
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("*");
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        ThreadPoolTaskScheduler taskScheduler = new ThreadPoolTaskScheduler();
        taskScheduler.setPoolSize(1);
        taskScheduler.setThreadNamePrefix("heartbeat-");
        taskScheduler.initialize();

        registry.enableSimpleBroker("/topic", "/queue")
                .setTaskScheduler(taskScheduler)
                .setHeartbeatValue(new long[]{10000, 10000});

        registry.setApplicationDestinationPrefixes("/app");
        registry.setUserDestinationPrefix("/user");
    }
}
```

То есть полностью убери этот метод:

```java
@Override
public void configureClientOutboundChannel(ChannelRegistration registration) {
    ...
}
```

После этого `connectAsync(...).get(...)` должен начать проходить.

## 4. Если frontend всё-таки нужен session id

Не отправляй его payload-ом внутри `CONNECTED`.

Лучше отправить отдельное обычное STOMP-сообщение после подключения или после первого запроса.

Например endpoint:

```java
@MessageMapping("/session")
public void sessionInfo(SimpMessageHeaderAccessor headerAccessor) {
    String sessionId = headerAccessor.getSessionId();
    // отправить ConnectedEvent обычным MESSAGE frame
}
```

Но если очень хочется именно header в `CONNECTED`, то interceptor должен быть максимально аккуратным.

Попробуй такой вариант:

```java
package com.pivotService.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.messaging.Message;
import org.springframework.messaging.MessageChannel;
import org.springframework.messaging.simp.SimpMessageHeaderAccessor;
import org.springframework.messaging.simp.SimpMessageType;
import org.springframework.messaging.simp.config.ChannelRegistration;
import org.springframework.messaging.simp.config.MessageBrokerRegistry;
import org.springframework.messaging.simp.stomp.StompHeaderAccessor;
import org.springframework.messaging.support.ChannelInterceptor;
import org.springframework.messaging.support.MessageBuilder;
import org.springframework.scheduling.concurrent.ThreadPoolTaskScheduler;
import org.springframework.web.socket.config.annotation.EnableWebSocket;
import org.springframework.web.socket.config.annotation.EnableWebSocketMessageBroker;
import org.springframework.web.socket.config.annotation.StompEndpointRegistry;
import org.springframework.web.socket.config.annotation.WebSocketMessageBrokerConfigurer;

@Configuration
@EnableWebSocketMessageBroker
@EnableWebSocket
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    public static final String CONNECTED_SESSION_ID_HEADER = "x-simp-session-id";

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOriginPatterns("*");
    }

    @Override
    public void configureClientOutboundChannel(ChannelRegistration registration) {
        registration.interceptors(new ChannelInterceptor() {

            @Override
            public Message<?> preSend(Message<?> message, MessageChannel channel) {
                SimpMessageType messageType = SimpMessageHeaderAccessor.getMessageType(message.getHeaders());

                if (!SimpMessageType.CONNECT_ACK.equals(messageType)) {
                    return message;
                }

                String sessionId = SimpMessageHeaderAccessor.getSessionId(message.getHeaders());

                if (sessionId == null || sessionId.isBlank()) {
                    return message;
                }

                StompHeaderAccessor accessor = StompHeaderAccessor.wrap(message);

                accessor.setLeaveMutable(true);
                accessor.setNativeHeader(CONNECTED_SESSION_ID_HEADER, sessionId);

                /*
                 * Критически важно:
                 * payload оставляем исходным.
                 * Никакой JSON в CONNECTED frame класть нельзя.
                 */
                return MessageBuilder.createMessage(
                        message.getPayload(),
                        accessor.getMessageHeaders()
                );
            }
        });
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        ThreadPoolTaskScheduler taskScheduler = new ThreadPoolTaskScheduler();
        taskScheduler.setPoolSize(1);
        taskScheduler.setThreadNamePrefix("heartbeat-");
        taskScheduler.initialize();

        registry.enableSimpleBroker("/topic", "/queue")
                .setTaskScheduler(taskScheduler)
                .setHeartbeatValue(new long[]{10000, 10000});

        registry.setApplicationDestinationPrefixes("/app");
        registry.setUserDestinationPrefix("/user");
    }
}
```

Но я бы сначала проверил тест **вообще без этого interceptor-а**.

## 6. Ещё одна важная вещь: destination

У тебя сейчас:

```java
private static final String DASHBOARD_MESSAGES_DESTINATION = "/user/topic/trades";
```

Это корректно только если `MessageSender` отправляет примерно так:

```java
simpMessagingTemplate.convertAndSendToUser(
        sessionId,
        "/topic/trades",
        payload,
        headers
);
```

И при этом headers содержат session id:

```java
headers.setSessionId(sessionId);
```

Например корректный `MessageSender` обычно выглядит так:

```java
package com.pivotService.controller.sender;

import lombok.RequiredArgsConstructor;
import org.springframework.messaging.simp.SimpMessageHeaderAccessor;
import org.springframework.messaging.simp.SimpMessagingTemplate;
import org.springframework.messaging.support.MessageHeaders;
import org.springframework.stereotype.Component;

@Component
@RequiredArgsConstructor
public class MessageSender {

    private final SimpMessagingTemplate messagingTemplate;

    public void sendToSession(String sessionId, String payload) {
        SimpMessageHeaderAccessor accessor = SimpMessageHeaderAccessor.create();

        accessor.setSessionId(sessionId);
        accessor.setLeaveMutable(true);

        MessageHeaders headers = accessor.getMessageHeaders();

        messagingTemplate.convertAndSendToUser(
                sessionId,
                "/topic/trades",
                payload,
                headers
        );
    }
}
```

Тогда клиентская подписка:

```java
"/user/topic/trades"
```

нормальная.

Если же `MessageSender` делает:

```java
simpMessagingTemplate.convertAndSend("/topic/trades", payload);
```

тогда тест должен подписываться на:

```java
private static final String DASHBOARD_MESSAGES_DESTINATION = "/topic/trades";
```

## 9. Что проверить следующим шагом

Сделай по порядку:

1. **Полностью убери `configureClientOutboundChannel()` из `WebSocketConfig`.**
2. Убери проверку `x-simp-session-id` из теста.
3. Запусти тест.
4. Если подключение прошло, но тест завис на ожидании `ConfigurationUpdated`, значит проблема уже не в WebSocket connect, а в:
   - destination;
   - отправке ответа;
   - сохранении сессии;
   - формате сообщения.
5. Если подключение всё равно падает, тогда нужно смотреть уже server-side exception именно в момент `CONNECT`.

Главный вывод: текущий `ConnectionLostException` возникает **до бизнес-логики**. Сначала нужно стабилизировать STOMP connect, а уже потом проверять `ConfigurationUpdated` и `ReceiveData`.

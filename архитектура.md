переведи на русский:
## Executive recommendation

Keep **Apache Pinot** as the fast OLAP snapshot and reconciliation store, but do **not** make Pinot polling the subscription engine.

Replace the current **A + Redis + B + SQL proxy** design with a **Subscription Query Fabric**:

1. **One real-time subscription API path**: the WebSocket/SSE gateway receives the subscription request, plans it, gets the first snapshot, and attaches the stream. Avoid “REST writes SQL to Redis, WebSocket later discovers it”.
2. **Canonical query ownership**: normalize every request into a canonical logical/physical plan and assign that plan to a sharded **Result Broker**. Many users/widgets sharing the same plan get one computed result.
3. **Snapshot path**: Result Broker serves the initial snapshot from active result state, cache, Pinot, or source adapters, with a strict deadline.
4. **Update path**: consume the same Kafka streams that feed Pinot, compute either:
   - incremental aggregate deltas for supported queries, or
   - dirty notifications that trigger coalesced Pinot re-queries.
5. **No topic per subscription**: use fixed sharded internal topics/streams keyed by `planId` or direct gRPC streams between Delta Engine shards and Result Broker shards.
6. **Multi-source queries**: avoid large runtime joins in Pinot. Prefer denormalization, fixed materializations, or joining bounded aggregate result sets outside Pinot in the Result Broker.

The critical change is this: **Pinot answers analytical reads; a separate subscription fabric owns fanout, caching, deltas, watermarks, and consistency.**

## 1. Evaluation of the current A + Redis + B + proxy design

| Area | Risk |
|---|---|
| REST service A stores SQL in Redis | Adds an extra control-plane hop on the critical path. The component that receives the subscription is not the component that can immediately serve the first snapshot. |
| Service B periodically polls Pinot | Polling millions of subscriptions is the main bottleneck. Even at one poll per second, $1{,}000{,}000$ subscriptions means $1{,}000{,}000$ Pinot queries/sec before deduplication. |
| Redis as subscription registry | Redis becomes a high-churn metadata bottleneck. It stores SQL strings, not semantic plans, watermarks, result state, subscriber fanout state, or query capabilities. |
| SQL-string proxy | A SQL proxy can deduplicate identical strings, but it cannot safely merge “similar” queries without semantic understanding, authorization context, grouping cardinality, and cost modeling. |
| Undefined change notifications | To trigger queries only when data changes, the system needs an event-to-query interest index. That cannot be solved reliably at the Pinot proxy layer alone. |
| First snapshot SLA | The path A → Redis → B → Pinot → B → client makes a hard $\le 100$ ms first snapshot SLA difficult. Polling makes it worse because the client can wait until the next poll tick. |
| Multi-source consistency | Separate polling of Pinot tables and other sources can produce skewed results: e.g. orders at one logical time, risk metrics at another. |
| Joins | Relying on runtime joins inside Pinot for subscription traffic creates unpredictable latency. Large fact-fact joins are especially risky. |
| Load smoothing | Spreading queries over time reduces Pinot spikes but directly trades off update latency. For active trader dashboards this becomes visible. |

Redis and a Pinot proxy are not useless, but they should not be the core subscription mechanism. Redis can remain as an optional distributed cache or coordination helper, and the proxy should evolve into a semantic **Query Serving Layer**, not a dumb SQL merger.

## 3. Main components and responsibilities

| Component | Responsibility |
|---|---|
| **Edge Stream Gateway** | Terminates WebSocket/SSE connections, authenticates users, multiplexes many widget subscriptions over one connection, sends snapshots/deltas to clients, handles backpressure. |
| **Planner / Semantic Query Service** | Validates frontend JSON, applies authorization predicates, canonicalizes query, generates physical fragments for Pinot and other sources, computes `planId`, performs admission control. Apache Calcite belongs here. |
| **Result Broker / Query Owner** | Owns each canonical plan by consistent hashing. Maintains active result, subscribers, sequence numbers, watermarks, cache, diffing, and update policy. New subscribers to an already-active plan get an immediate snapshot from memory. |
| **Snapshot Executor** | Executes physical snapshot fragments against Pinot and source adapters in parallel with deadlines. Merges bounded result sets. |
| **Delta Engine** | Consumes canonical Kafka streams, maps events to affected plans, computes incremental aggregate deltas where possible, otherwise emits dirty notifications. |
| **Watermark Service** | Tracks logical progress of Pinot ingestion, Delta Engine consumption, and other source adapters. Provides consistent `asOf` vectors. |
| **Source Adapters** | Provide snapshot and delta capabilities for non-Pinot sources such as risk state stores, portfolio valuation services, or computed metric stores. |
| **Pinot** | Fast OLAP store for snapshots, rollups, historical queries, reconciliation, and cold fallback. |
| **Optional distributed cache** | Stores serialized result snapshots keyed by `planId` and watermark. Redis/KeyDB/Memcached can be used here, but not as the primary subscription registry. |

## 4. Subscription and first snapshot flow

```text
1. Client sends SubscribeRequest over WebSocket.

2. Gateway authenticates user and forwards request to Planner.

3. Planner:
   - validates JSON DSL,
   - injects tenant/user authorization filters,
   - resolves semantic measures/dimensions,
   - builds canonical logical plan,
   - builds physical fragments,
   - computes planId,
   - checks whether the plan is eligible for real-time subscription.

4. Gateway routes the subscription to the Result Broker that owns planId.

5. Result Broker:
   - if plan already active: returns current in-memory snapshot immediately;
   - else if result cache hit: returns cached snapshot and starts update tracking;
   - else executes a high-priority snapshot via Snapshot Executor.

6. Snapshot Executor queries Pinot and source adapters in parallel.

7. Result Broker stores result state, assigns seq=1, attaches subscriber, sends snapshot.

8. Delta Engine receives plan registration if this is the first active subscriber for planId.
```

A typical external response shape:

```json
{
  "type": "snapshot",
  "subscriptionId": "widget-123",
  "seq": 1,
  "asOf": {
    "orders": { "topic": "orders_canonical", "offsetWatermark": "..." },
    "risk": { "topic": "risk_metrics", "offsetWatermark": "..." }
  },
  "schema": [
    { "name": "portfolioId", "type": "string" },
    { "name": "symbol", "type": "string" },
    { "name": "openNotional", "type": "decimal" },
    { "name": "var95", "type": "decimal" }
  ],
  "rows": [
    {
      "key": ["P1", "AAPL"],
      "portfolioId": "P1",
      "symbol": "AAPL",
      "openNotional": 1250000.50,
      "var95": 92000.00
    }
  ]
}
```

### Important SLA point

A hard first snapshot SLA of $\le 100$ ms should be defined for an **admitted serving query class**, not arbitrary analytical SQL over months of raw data.

The subscription API should accept only plans that have at least one of these:

1. Active Result Broker state.
2. Hot result cache.
3. Precomputed Pinot rollup/star-tree path with measured $p99$ below the deadline.
4. Incremental aggregate state.
5. A bounded source-adapter point-read path.

For non-admitted exploratory queries, expose a separate async “explore/query” API, not the live subscription API.

## 5. Continuous update mechanism

Do **not** poll Pinot per subscription.

Use a hybrid push design:

1. **Incremental push** for supported aggregates.
2. **Event-triggered dirty re-query** for complex queries.
3. **Periodic reconciliation** against Pinot to correct drift and recover from missed events.

### 5.1 Canonical event envelope

All sources should be normalized into a common envelope before feeding Pinot and the Delta Engine.

```json
{
  "source": "postgres",
  "entity": "orders",
  "op": "UPSERT",
  "tenantId": "T1",
  "primaryKey": "order-987",
  "eventTimeMs": 1760000000123,
  "ingestTimeMs": 1760000000170,
  "version": {
    "kind": "postgres_lsn",
    "value": "16/B374D848"
  },
  "partition": 42,
  "offset": 918273645,
  "before": {
    "status": "OPEN",
    "remainingQty": 100,
    "price": 10.0,
    "portfolioId": "P1",
    "symbol": "AAPL"
  },
  "after": {
    "status": "PARTIAL",
    "remainingQty": 50,
    "price": 10.0,
    "portfolioId": "P1",
    "symbol": "AAPL"
  }
}
```

For correct aggregate deltas on updates/deletes, the Delta Engine needs either:

- Debezium `before` images, or
- a previous-value state store keyed by source primary key.

### 5.2 Dirty notification path

For queries that are not incrementally maintainable, the Delta Engine does not compute the result. It only determines that a canonical plan is dirty.

```text
Kafka event batch
    → compute affected descriptors: tenant, source, symbol, portfolio, time bucket, entity key
    → consult subscription interest indexes
    → emit Dirty(planId, affectedKeys, sourceOffsetRange)
    → Result Broker coalesces dirty signals for a small interval
    → Result Broker re-queries Pinot/source adapters at a consistent watermark
    → Result Broker diffs old result vs new result
    → Gateway pushes delta to clients
```

The internal dirty stream is fixed and sharded:

```text
dashboard.dirty.000 ... dashboard.dirty.1023
```

Messages are keyed by `planId`. There is no topic per subscription.

### 5.3 Incremental aggregate path

For common trader dashboard widgets, maintain deltas directly from streams.

Good candidates:

- `SUM`
- `COUNT`
- `MIN` / `MAX`, with caveats for deletes
- `AVG`, as `SUM` plus `COUNT`
- latest value by key
- simple grouped time buckets
- some sketches such as HLL or TDigest, if approximation is acceptable

Example:

```text
Order update:
  old contribution: portfolio=P1, symbol=AAPL, openNotional=1000
  new contribution: portfolio=P1, symbol=AAPL, openNotional=500

Delta:
  group(P1,AAPL).openNotional += -500
```

The Result Broker receives an aggregate delta and updates its current result state without hitting Pinot.

### 5.4 Recommended hybrid policy

| Query type | Update strategy |
|---|---|
| Hot additive dashboard aggregates | Incremental Delta Engine |
| Latest risk metric by portfolio | Source adapter/latest-state delta |
| Percentiles/top-N/sketches | Incremental sketch if acceptable, otherwise dirty re-query |
| Large historical rollups | Event-triggered Pinot re-query with coalescing |
| Complex multi-source derived result | Incremental fragments plus broker-side recomputation, or dirty re-query |
| Low-QPS/cold widgets | Dirty re-query only |
| Suspicious drift/gaps | Reconciliation snapshot from Pinot |

This gives low latency for common widgets without requiring a materialized view or Kafka topic per subscription.

## 6. Multi-source query strategy

Use a **semantic planner** and compose the response from source fragments.

A single frontend query should become a physical plan like:

```text
CompositePlan planId=abc123

Fragment A:
  source = Pinot.orders_latest
  output = GROUP BY portfolioId, symbol
  measures = SUM(openNotional), COUNT(orderId)

Fragment B:
  source = RiskLatestAdapter
  output = latest var95 by portfolioId

Fragment C:
  source = Pinot.quotes_latest
  output = latest mid by symbol

Merge:
  join A with B on portfolioId
  join with C on symbol
  compute derived fields
```

### Preferred join hierarchy

| Join type | Recommended implementation |
|---|---|
| Fact + slowly changing dimensions | Denormalize/enrich before Pinot ingestion. |
| Aggregate result + aggregate result | Join outside Pinot in the Result Broker. Result sets are bounded. |
| Latest metric lookup | Use a latest-state store or Pinot upsert table, then join outside Pinot. |
| Large fact-fact join | Precompute/materialize into a serving table. Do not do this dynamically in the subscription path. |
| Small dimension broadcast join | Pinot multi-stage engine may be acceptable after benchmarking. |
| Ad hoc analytical join | Use Pinot multi-stage engine outside the hard real-time subscription SLA. |

### Should Pinot multi-stage query engine be used?

Use it selectively.

For the live subscription path, prefer:

1. Pre-enriched Pinot fact tables.
2. Broker-side joins of small aggregate result sets.
3. Fixed materializations for large joins.

Pinot multi-stage joins are useful, but they add distributed exchange, more variable latency, and more difficult cost control. They are better for controlled, benchmarked cases than for arbitrary high-churn subscriptions.

## 7. Pinot data organization

### 7.1 Table categories

Use several serving tables rather than one universal table.

| Table type | Example | Purpose |
|---|---|---|
| Append event tables | `orders_events_rt`, `transactions_rt` | Historical event analytics. |
| Latest/upsert tables | `orders_latest_rt`, `positions_latest_rt`, `quotes_latest_rt` | Current-state dashboards. |
| Metric tables | `risk_metrics_rt`, `portfolio_value_rt` | Risk and valuation snapshots/time series. |
| Rollup tables | `orders_agg_1s`, `orders_agg_1m`, `transactions_agg_1m` | Fast grouped historical queries. |
| Cuboid tables | `portfolio_symbol_exposure_1s`, `desk_asset_risk_1m` | High-volume dashboard templates. |
| Dimension tables | `instrument_dim`, `account_dim`, `portfolio_dim` | Mostly for enrichment or small joins. |

### 7.2 General schema principles

Every serving table should include:

- `tenant_id`
- event/logical time column, e.g. `event_ts`
- ingestion time, e.g. `ingest_ts`
- source version, e.g. Postgres LSN or source sequence
- business dimensions used in filters/groupings
- precomputed measure columns where possible
- deletion/tombstone flags for upsert tables

Example columns for `orders_latest_rt`:

```text
tenant_id
order_id
portfolio_id
account_id
desk_id
symbol
asset_class
side
status
remaining_qty
limit_price
open_notional
event_ts
source_lsn
is_deleted
```

### 7.3 Indexing

Recommended Pinot indexes:

| Index | Use |
|---|---|
| Sorted index | Commonly filtered time or tenant/time columns. |
| Inverted index | Low/medium-cardinality dimensions such as desk, status, asset class, venue. |
| Range index | Numeric and time filters. |
| Bloom filter | High-cardinality equality lookup columns where needed. |
| Star-tree | Repeated aggregate dashboards with stable grouping dimensions. |
| JSON index | Only for controlled semi-structured fields. Avoid for core high-QPS paths. |
| HLL/Theta/TDigest columns | Approximate distinct counts or percentiles where accepted. |

### 7.4 Partitioning

Partition by dimensions that are present in nearly every query:

```text
tenant_id
tenant_id + portfolio_id
tenant_id + account_id
tenant_id + symbol
```

The exact choice depends on query patterns. For trader dashboards, `tenant_id` plus portfolio/account/symbol is usually more useful than partitioning only by technical primary key.

For upsert tables, Pinot primary-key partitioning requirements may conflict with query pruning. In those cases, keep multiple projections:

```text
orders_latest_by_order_id_rt       -- upsert-correctness projection
orders_exposure_by_portfolio_rt    -- query-serving projection
orders_exposure_by_symbol_rt       -- query-serving projection
```

### 7.5 Rollups

Use rollup tables for common time ranges:

| Time range | Serving table |
|---|---|
| Last few minutes | real-time latest/event table or 1-second rollup |
| Intraday | 1-second or 10-second rollup |
| Days/weeks | 1-minute rollup |
| Months | hourly/daily rollup |

The Planner rewrites the query to the smallest table that satisfies the requested accuracy.

## 8. Consistency model

The dashboard should show a result that is consistent **as of a declared logical point**, not a mixture of unrelated source times.

### 8.1 Use version vectors

Each snapshot and delta should carry an `asOf` vector:

```json
{
  "orders": {
    "topic": "orders_canonical",
    "partitions": {
      "0": 88123,
      "1": 99111
    }
  },
  "risk": {
    "topic": "risk_metrics",
    "partitions": {
      "0": 55102
    }
  }
}
```

The Result Broker should only move a subscription from one `asOf` vector to a later one.

### 8.2 Align sources

For multi-source results, choose one of these policies:

| Policy | Behavior |
|---|---|
| Strict aligned watermark | Result advances only when all required sources have reached a common logical point. Highest consistency, more latency. |
| Source-vector consistency | Result carries independent source offsets. Good balance. |
| Stale-source marking | If risk lags orders, show latest orders but mark risk as stale. Useful when UX accepts it. |

For trading dashboards, the safest default is source-vector consistency with explicit stale markers.

### 8.3 Avoid visible aggregate jumps

To avoid jumps caused by uncoordinated updates:

- apply Postgres transaction metadata from Debezium where possible;
- process update/delete retractions atomically;
- do not emit a multi-source derived row until all required fragments are available for the target watermark;
- sequence all messages per subscr

## Главный вывод и рекомендация

**Apache Pinot** оставляем как быстрое OLAP-хранилище для снапшотов и перестроений, но **не делаем опрос Pinot основой механизма подписок**.

Вместо цепочки **A + Redis + B + SQL-прокси** предлагаю **фабрику подписок** (Subscription Query Fabric):

1. **Единый путь создания real-time подписки**: шлюз (WebSocket/SSE) получает запрос, строит план, берёт первый снапшот и сразу подключает поток обновлений. Никакой схемы «REST пишет SQL в Redis, а WebSocket потом это читает».
2. **Канонические запросы**: каждый входящий запрос нормализуется в канонический логический/физический план, и этот план «принадлежит» конкретному шардированному **Result Broker**. Если один и тот же план нужен нескольким пользователям или виджетам, он вычисляется один раз, а результат шарится.
3. **Снапшотный путь**: Result Broker выдаёт начальное состояние из активного результата (если план уже работает), из кэша, из Pinot или из адаптеров источников, строго соблюдая дедлайн.
4. **Путь обновлений**: потребляем те же Kafka-топики, что питают Pinot, и считаем:
   - либо **инкрементальные дельты** для поддерживаемых агрегатов,
   - либо **помечаем план «грязным»** (dirty), после чего по триггеру перезапрашиваем Pinot.
5. **Никакого топика на каждую подписку**: используем фиксированный пул шардированных внутренних топиков/стримов, ключом является `planId`, либо прямое gRPC-соединение между шардами Delta Engine и Result Broker.
6. **Мульти-источниковые запросы**: избегаем тяжёлых динамических join'ов внутри Pinot. Предпочитаем денормализацию, заранее материализованные представления, либо объединение небольших агрегированных наборов на стороне Result Broker.

Ключевое изменение: **Pinot отвечает за аналитические чтения; отдельная фабрика подписок владеет фан-аутом, кэшем, дельтами, версионностью и консистентностью.**

## 1. Оценка текущей конструкции (A + Redis + B + прокси)

| Проблемная область | Риск |
|---|---|
| REST-сервис A сохраняет SQL в Redis | Добавляется лишний хоп на критическом пути. Компонент, принимающий подписку, не является тем, кто тут же может отдать первый снапшот. |
| Сервис B периодически опрашивает Pinot | Опрос миллионов подписок — основной бутылочное горлышко. Даже при 1 опросе в секунду, 1 000 000 подписок = 1 000 000 запросов в секунду до всякой дедупликации. |
| Redis как реестр подписок | Redis превращается в место с высокой скоростью обновления метаданных. Хранятся строки SQL, а не семантические планы, водяные знаки (watermarks), состояния результатов, списки подписчиков или возможности запросов. |
| Прокси на основе SQL-строк | Прокси умеет дедуплицировать идентичные строки, но не может безопасно объединять «похожие» запросы без понимания семантики, контекста прав, кардинальности группировки и стоимости. |
| Отсутствие механизма уведомлений об изменениях | Чтобы дёргать запросы только при появлении данных, нужен индекс интересов (event-to-query interest index). Надёжно построить его только в слое прокси перед Pinot нельзя. |
| SLA первого снапшота | Путь A → Redis → B → Pinot → B → клиент делает жёсткое ≤100 мс крайне труднодостижимым. Опрос усугубит ситуацию — клиент может ждать следующего тика опроса. |
| Консистентность при нескольких источниках | Раздельный опрос таблиц Pinot и других источников может давать несогласованные результаты: например, ордера на один логический момент, а метрики риска на другой. |
| Join'ы | Использование динамических join'ов внутри Pinot для трафика подписок приводит к непредсказуемой задержке. Особенно опасны join'ы крупных факт-таблиц. |
| Сглаживание нагрузки | Равномерное распределение запросов во времени снижает пики на Pinot, но прямо увеличивает задержку обновлений. Для активных трейдерских дашбордов это заметно. |

Redis и прокси не бесполезны, но им не место в ядре механизма подписок. Redis может остаться опциональным распределённым кэшем или координатором, а прокси должно превратиться в **семантический слой обслуживания запросов (Query Serving Layer)**, а не в тупой мержер SQL-строк.

## 3. Основные компоненты и их обязанности

| Компонент | Обязанность |
|---|---|
| **Edge Stream Gateway** | Завершает WebSocket/SSE соединения, аутентифицирует пользователя, мультиплексирует множество подписок виджетов в одном соединении, отправляет клиенту снапшоты и дельты, управляет обратным давлением. |
| **Planner / Семантический сервис запросов** | Валидирует входящий JSON, применяет предикаты авторизации (например, tenant), нормализует запрос, строит физические фрагменты для Pinot и других источников, считает `planId`, контролирует допуск (admission control). Здесь ваш Apache Calcite. |
| **Result Broker / Владелец запроса** | Владеет каждым каноническим планом (через consistent hashing). Хранит активный результат, подписчиков, порядковые номера, водяные знаки, кэш, умеет вычислять дельты и выбирать стратегию обновления. Новые подписчики на уже активный план сразу получают снапшот из памяти. |
| **Snapshot Executor** | Выполняет фрагменты физического плана против Pinot и адаптеров источников параллельно, соблюдая deadline. Объединяет ограниченные результирующие наборы. |
| **Delta Engine** | Потребляет канонические Kafka-стримы, сопоставляет события с затронутыми планами, вычисляет инкрементальные агрегатные дельты (где возможно), иначе генерирует dirty-уведомления. |
| **Watermark Service** | Отслеживает логический прогресс поставки в Pinot, потребления Delta Engine и адаптеров. Предоставляет согласованные векторы `asOf`. |
| **Source Adapters** | Обеспечивают снапшот и дельта-возможности для не-Pinot источников: хранилищ риск-метрик, сервисов оценки портфеля и т.п. |
| **Pinot** | Быстрое OLAP-хранилище для снапшотов, роллапов, исторических запросов, проверки и холодного восстановления. |
| **Опциональный распределённый кэш** | Хранит сериализованные снапшоты с ключом `planId` и водяным знаком. Redis/KeyDB/Memcached могут применяться здесь, но не как первичный реестр подписок. |

---

## 4. Поток создания подписки и получения первого снапшота

```text
1. Клиент отправляет SubscribeRequest через WebSocket.

2. Шлюз аутентифицирует пользователя и передаёт запрос Planner.

3. Planner:
   - валидирует JSON DSL,
   - добавляет фильтры арендатора/пользователя,
   - разрешает семантические меры и измерения,
   - строит канонический логический план,
   - строит физические фрагменты (для Pinot и адаптеров),
   - вычисляет planId,
   - проверяет, допустим ли план для real-time подписки.

4. Шлюз маршрутизирует подписку к Result Broker, владеющему planId.

5. Result Broker:
   - если план уже активен: немедленно отдаёт текущий снапшот из памяти;
   - иначе если есть попадание в кэш: возвращает кэшированный снапшот и начинает отслеживание обновлений;
   - иначе выполняет приоритетный снапшот через Snapshot Executor.

6. Snapshot Executor параллельно опрашивает Pinot и адаптеры источников.

7. Result Broker сохраняет состояние результата, присваивает seq=1, подключает подписчика, отправляет снапшот.

8. Если это первый активный подписчик на planId, Delta Engine получает регистрацию плана.
```

Пример внешнего ответа:

```json
{
  "type": "snapshot",
  "subscriptionId": "widget-123",
  "seq": 1,
  "asOf": {
    "orders": { "topic": "orders_canonical", "offsetWatermark": "..." },
    "risk": { "topic": "risk_metrics", "offsetWatermark": "..." }
  },
  "schema": [...],
  "rows": [
    {
      "key": ["P1", "AAPL"],
      "portfolioId": "P1",
      "symbol": "AAPL",
      "openNotional": 1250000.50,
      "var95": 92000.00
    }
  ]
}
```

### Важное замечание о SLA

Жёсткий SLA ≤100 мс для первого снапшота должен действовать только для **допущенного класса обслуживаемых запросов**, а не для произвольного аналитического SQL по данным за месяцы. API подписок должно принимать только планы, удовлетворяющие хотя бы одному из условий:

1. Активное состояние в Result Broker.
2. Попадание в горячий кэш.
3. Предварительно вычисленный роллап/star-tree в Pinot с измеренным p99 ниже дедлайна.
4. Состояние инкрементального агрегата.
5. Ограниченный путь point-read через адаптер источника.

Для недопущенных исследовательских запросов используйте отдельный асинхронный API «explore/query», не live-подписку.

## 5. Механизм непрерывных обновлений

**Никакого опроса Pinot на каждую подписку.** Используем гибридный push-подход:

1. **Инкрементальный push** для поддерживаемых агрегатов.
2. **Триггерный грязный перезапрос** (dirty re-query) для сложных запросов.
3. **Периодическая сверка** с Pinot для коррекции дрейфа и восстановления после пропущенных событий.

### 5.1 Канонический конверт события

Все источники должны нормализоваться в общий конверт перед тем, как попасть в Pinot и Delta Engine.

```json
{
  "source": "postgres",
  "entity": "orders",
  "op": "UPSERT",
  "tenantId": "T1",
  "primaryKey": "order-987",
  "eventTimeMs": 1760000000123,
  "ingestTimeMs": 1760000000170,
  "version": { "kind": "postgres_lsn", "value": "16/B374D848" },
  "partition": 42,
  "offset": 918273645,
  "before": {
    "status": "OPEN", "remainingQty": 100, "price": 10.0,
    "portfolioId": "P1", "symbol": "AAPL"
  },
  "after": {
    "status": "PARTIAL", "remainingQty": 50, "price": 10.0,
    "portfolioId": "P1", "symbol": "AAPL"
  }
}
```

Для корректного расчёта дельт при обновлениях/удалениях Delta Engine нуждается либо в Debezium `before`-образах, либо в собственном стейт-сторе с предыдущими значениями по первичному ключу источника.

### 5.2 Путь dirty-уведомлений

Для запросов, которые невозможно обслуживать инкрементально, Delta Engine не вычисляет результат, а лишь определяет, что канонический план стал «грязным».

```text
Пакет событий Kafka
   → вычисляем затронутые дескрипторы: tenant, source, symbol, portfolio, временной интервал, ключ сущности
   → сверяемся с индексами интересов подписок
   → генерируем Dirty(planId, affectedKeys, sourceOffsetRange)
   → Result Broker аккумулирует dirty-сигналы за короткий интервал
   → Result Broker перезапрашивает Pinot/адаптеры на согласованном водяном знаке
   → Result Broker сравнивает старый и новый результат
   → Шлюз пушит клиенту дельту
```

Внутренний стрим dirty-уведомлений фиксирован и шардирован:

```text
dashboard.dirty.000 ... dashboard.dirty.1023
```

Сообщения ключаются по `planId`. На каждую подписку свой топик не создаётся.

### 5.3 Инкрементальные агрегаты

Для наиболее распространённых виджетов трейдеров поддерживаем прямые дельты из стримов.

Хорошие кандидаты:

- `SUM`
- `COUNT`
- `MIN` / `MAX` (с оговорками для удалений)
- `AVG` как `SUM` + `COUNT`
- последнее значение по ключу
- простые группировки по временным окнам
- некоторые аппроксимационные скечи (HLL, TDigest), если допустима приблизительность

Пример:

```text
Обновление ордера:
  старый вклад: portfolio=P1, symbol=AAPL, openNotional=1000
  новый вклад: portfolio=P1, symbol=AAPL, openNotional=500

Дельта:
  group(P1,AAPL).openNotional += -500
```

Result Broker применяет агрегатную дельту и обновляет текущий результат без обращения к Pinot.

### 5.4 Рекомендуемая гибридная политика

| Тип запроса | Стратегия обновления |
|---|---|
| Горячие аддитивные агрегаты дашбордов | Инкрементальный Delta Engine |
| Последнее значение риск-метрики по портфелю | Адаптер источника / latest-state дельта |
| Процентили/top-N/скечи | Инкрементальный скеч если допустимо, иначе dirty re-query |
| Большие исторические роллапы | Триггерный перезапрос Pinot с объединением |
| Сложные мульти-источниковые производные | Инкрементальные фрагменты и пересчёт на брокере, либо dirty re-query |
| Низкочастотные/«холодные» виджеты | Только dirty re-query |
| Обнаруженный дрейф/пропуски | Сверочный снапшот из Pinot |

Это обеспечивает низкую задержку для частых виджетов без необходимости в материализованном представлении или Kafka-топике на каждую подписку.

## 6. Стратегия мульти-источниковых запросов

Используем **семантический планировщик**, а ответ компонуем из фрагментов разных источников.

Один запрос с фронтенда должен превращаться в физический план вроде:

```text
CompositePlan planId=abc123

Фрагмент A:
  источник = Pinot.orders_latest
  вывод = GROUP BY portfolioId, symbol
  меры = SUM(openNotional), COUNT(orderId)

Фрагмент B:
  источник = RiskLatestAdapter
  вывод = последнее var95 по portfolioId

Фрагмент C:
  источник = Pinot.quotes_latest
  вывод = последнее mid по symbol

Объединение:
  join A с B по portfolioId
  join с C по symbol
  вычисление производных полей
```

### Предпочтительная иерархия join'ов

| Тип join'а | Рекомендуемая реализация |
|---|---|
| Факт + медленно меняющееся измерение | Денормализовать/обогатить до загрузки в Pinot. |
| Агрегатный результат + агрегатный результат | Join вне Pinot, в Result Broker. Результаты заведомо ограничены. |
| Поиск последней метрики | Хранилище latest-state или upsert-таблица Pinot, join снаружи. |
| Крупный fact-fact join | Предвычислить и материализовать в обслуживающую таблицу. Не делать динамически в пути подписки. |
| Broadcast join с маленьким измерением | Можно попробовать multi-stage engine Pinot после бенчмаркинга. |
| Ad hoc аналитический join | Использовать multi-stage engine Pinot вне жёсткого real-time SLA подписки. |

### Стоит ли использовать Pinot multi-stage query engine?

Выборочно. Для live-подписок предпочтительнее:

1. Пре-обогащённые факт-таблицы Pinot.
2. Join'ы небольших агрегатных результатов на стороне брокера.
3. Фиксированные материализации для крупных join'ов.

Pinot multi-stage join'ы полезны, но они добавляют распределённый обмен, увеличивают разброс задержек и усложняют контроль стоимости. Они лучше для контролируемых, оттестированных случаев, чем для произвольных высоко-изменчивых подписок.

## 7. Организация данных в Pinot

### 7.1 Категории таблиц

Используем несколько обслуживающих таблиц, а не одну универсальную.

| Тип таблицы | Пример | Назначение |
|---|---|---|
| Append-таблицы событий | `orders_events_rt`, `transactions_rt` | Историческая аналитика событий. |
| Latest/upsert таблицы | `orders_latest_rt`, `positions_latest_rt`, `quotes_latest_rt` | Дашборды текущего состояния. |
| Таблицы метрик | `risk_metrics_rt`, `portfolio_value_rt` | Снапшоты рисков и оценок. |
| Роллап-таблицы (сводные) | `orders_agg_1s`, `orders_agg_1m`, `transactions_agg_1m` | Быстрые групповые исторические запросы. |
| Кубоидные таблицы | `portfolio_symbol_exposure_1s`, `desk_asset_risk_1m` | Шаблонные высоко-объёмные дашборды. |
| Таблицы измерений | `instrument_dim`, `account_dim`, `portfolio_dim` | Для обогащения или маленьких join'ов. |

### 7.2 Общие принципы схемы

Каждая обслуживающая таблица должна содержать:

- `tenant_id`
- колонку событийного/логического времени, например `event_ts`
- время инжеста, например `ingest_ts`
- версию источника, например Postgres LSN или последовательность источника
- бизнес-измерения для фильтрации/группировки
- предвычисленные колонки мер, где возможно
- флаги удаления/tombstone для upsert-таблиц

Пример колонок для `orders_latest_rt`:

```text
tenant_id
order_id
portfolio_id
account_id
desk_id
symbol
asset_class
side
status
remaining_qty
limit_price
open_notional
event_ts
source_lsn
is_deleted
```

### 7.3 Индексация

Рекомендуемые индексы Pinot:

| Индекс | Применение |
|---|---|
| Sorted index | Часто фильтруемые колонки времени или tenant/time. |
| Inverted index | Измерения низкой/средней кардинальности: desk, status, asset class, venue. |
| Range index | Числовые и временные фильтры. |
| Bloom filter | Колонки с высокой кардинальностью для точечных запросов. |
| Star-tree | Повторяющиеся агрегатные дашборды со стабильным набором измерений. |
| JSON index | Только для контролируемых полуструктурированных полей. Избегать на высоконагруженных путях. |
| HLL/Theta/TDigest колонки | Приблизительный distinct count или процентили, где допустимо. |

### 7.4 Партиционирование

Партиционируем по измерениям, присутствующим почти в каждом запросе:

```text
tenant_id
tenant_id + portfolio_id
tenant_id + account_id
tenant_id + symbol
```

Конкретный выбор зависит от паттернов запросов. Для трейдерских дашбордов `tenant_id` + portfolio/account/symbol обычно полезнее, чем только технический первичный ключ.

Для upsert-таблиц требования первичного ключа Pinot могут конфликтовать с пропусканием партиций при запросе. В таких случаях используем несколько проекций:

```text
orders_latest_by_order_id_rt       -- проекция для корректности upsert
orders_exposure_by_portfolio_rt    -- обслуживающая запросы проекция
orders_exposure_by_symbol_rt       -- обслуживающая запросы проекция
```

### 7.5 Роллапы

Для типичных временных окон применяем роллап-таблицы:

| Временной диапазон | Обслуживающая таблица |
|---|---|
| Последние минуты | real-time latest/event таблица или 1-секундный роллап |
| Внутри дня | 1-секундный или 10-секундный роллап |
| Дни/недели | 1-минутный роллап |
| Месяцы | часовой/дневной роллап |

Planner переписывает запрос к минимальной таблице, удовлетворяющей требуемой точности.

## 8. Модель консистентности

Дашборд должен показывать результат, согласованный **на определённый логический момент**, а не смесь из несвязанных времён источников.

### 8.1 Вектора версий

Каждый снапшот и дельта должны нести вектор `asOf`:

```json
{
  "orders": {
    "topic": "orders_canonical",
    "partitions": { "0": 88123, "1": 99111 }
  },
  "risk": {
    "topic": "risk_metrics",
    "partitions": { "0": 55102 }
  }
}
```

Result Broker должен продвигать подписку только от одного вектора `asOf` к другому, более позднему.

### 8.2 Выравнивание источников

Для мульти-источниковых результатов выбираем одну из политик:

| Политика | Поведение |
|---|---|
| Строго выровненный водяной знак | Результат продвигается, только когда все требуемые источники достигли общей точки. Максимальная консистентность, больше задержка. |
| Консистентность векторов источников | Результат несёт независимые смещения источников. Хороший баланс. |
| Помечивание отстающих источников | Если риск отстаёт от ордеров, показываем последние ордера, но помечаем риск как устаревший. Использовать, когда UX это допускает. |

Для трейдинга безопасная по умолчанию — консистентность векторов источников с явными stale-маркерами.

### 8.3 Избежание видимых скачков агрегатов

Чтобы избежать скачков из-за нескоординированных обновлений:

- используем метаданные транзакций Postgres из Debezium, где возможно;
- атомарно обрабатываем ретракции обновлений/удалений;
- не выдаём производную строку по нескольким источникам, пока все фрагменты не готовы для целевого водяного знака;
- все сообщения последовательно нумеруются для каждой подписки.

## 9. Масштабирование и обработка миллионов подписок

### 9.1 Масштабирование шлюза и брокеров

- **Edge Gateway**: горизонтально масштабируется по соединениям. HPA по CPU и памяти.
- **Planner**: stateless, горизонтально масштабируется.
- **Result Broker**: шардируется по `planId` (consistent hashing). Каждый шард владеет группой планов, поддерживает in-memory стейт и фан-аут. Новые подписчики на существующий план — операция O(1) без дополнительной координации.
- **Delta Engine**: консьюмер-группа Kafka с шардированием по партициям источника. Индекс интересов — in-memory структура, синхронизируемая при изменениях активных планов.
- **Snapshot Executor**: stateless worker'ы.

### 9.2 Индекс интересов

Delta Engine строит индекс «событие → затронутые planId». При получении события он быстро находит все планы, которые нужно пометить грязными или обновить дельтой. Индекс кардинальности: количество уникальных планов на порядки меньше количества событий, поэтому он компактен.

### 9.3 Подписки с одинаковыми планами

Если множество пользователей/виджетов имеют идентичный канонический план (одинаковые фильтры, группировки), Result Broker хранит единственный активный результат и список подписчиков. Создание 100 000 подписок на такой же план требует лишь добавления 100 000 записей в список фан-аута, без дополнительных запросов к Pinot и без новых топиков.

## 10. Отказоустойчивость и восстановление

### 10.1 Сбои Result Broker

При падении шарда Result Broker его подписчики переподключаются к новому владельцу (согласно consistent hashing). Восстановление состояния:

- Из кэша снапшотов (если есть).
- Или быстрое перестроение через Pinot и догон из Delta Engine.

Delta Engine сохраняет прогресс в Kafka (consumer offsets) и может выдать пропущенные сообщения новому Result Broker.

### 10.2 Сбои Delta Engine

Перезапуск из offsets Kafka плюс пересинхронизация индекса интересов из реестра активных планов. Кратковременное отставание не вызывает потери данных, а лишь увеличивает задержку.

### 10.3 Сбои Pinot

При недоступности Pinot Result Broker обслуживает подписки из своего текущего состояния или помечает данные как stale. При восстановлении Pinot запускается сверка.

```text
PostgreSQL WAL → Debezium → сырые Kafka-топики
                               ↓

                               ↓
             канонические Kafka-топики
                    ┌──────────┴──────────┐

                  Pinot            Delta Engine
                    ↑                     ↓
             Snapshot Executor ←──Result Broker──→ Gateway → клиенты (WebSocket)
                    ↑                     ↑
            Source Adapters        Watermark Service
```

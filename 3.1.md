Да, в данном случае это **не должна быть ошибка**.

Если у двух классов поле выглядит так:

```text
Trade.side          jsonName=side type=STRING
TradeSaveEvent.side jsonName=side type=STRING
```

то для SQL это **одно и то же поле**:

```sql
"side"
```

Текущая ошибка появилась из-за слишком строгой проверки в `BusinessMetadataRegistry`: скорее всего, метод `sameFieldDefinition` сравнивает еще и `restrictions`. Поэтому поля визуально одинаковые, но валидатор считает их разными.

---

## Как исправить

Для SQL-конфига поле должно считаться неоднозначным только если отличается:

- `jsonName`, то есть имя колонки в SQL;
- `kind`, то есть тип поля.

Например, это **не ambiguity**:

```text
Trade.side          jsonName=side type=STRING
TradeSaveEvent.side jsonName=side type=STRING
```

А это уже **ambiguity**:

```text
Trade.side          jsonName=side type=STRING
SomeOther.side      jsonName=side type=LONG
```

или:

```text
Trade.side          jsonName=side type=STRING
SomeOther.side      jsonName=tradeSide type=STRING
```

---

# Быстрый фикс

В `BusinessMetadataRegistry` замените метод `sameFieldDefinition`.

## Было

```java
private static boolean sameFieldDefinition(FieldMetadata left, FieldMetadata right) {
    return Objects.equals(left.name(), right.name())
            && Objects.equals(left.jsonName(), right.jsonName())
            && left.kind() == right.kind()
            && Objects.equals(left.field().restrictions(), right.field().restrictions());
}
```

## Должно быть

```java
private static boolean sameFieldDefinition(FieldMetadata left, FieldMetadata right) {
    return Objects.equals(left.jsonName(), right.jsonName())
            && left.kind() == right.kind();
}
```

После этого поля:

```text
Trade.side jsonName=side type=STRING
TradeSaveEvent.side jsonName=side type=STRING
```

будут считаться совместимыми.

---

# Также лучше улучшить сообщение об ошибке

Сейчас сообщение говорит:

```text
Found different field definitions: Trade.side jsonName=side type=STRING; TradeSaveEvent.side jsonName=side type=STRING
```

Но визуально они одинаковые. Значит, различие было в чем-то, что не попало в описание, например в `restrictions`.

Замените `describeCandidate` на более подробный:

```java
private static String describeCandidate(FieldMetadata field) {
    return field.declaredInClass() +
            "." +
            field.name() +
            " jsonName=" +
            field.jsonName() +
            " kind=" +
            field.kind() +
            " rawType=" +
            field.rawType() +
            " restrictions=" +
            field.field().restrictions();
}
```

Тогда, если проблема действительно в `restrictions`, вы это сразу увидите.

# Более аккуратный вариант через SQL-сигнатуру

Еще лучше явно ввести понятие SQL-сигнатуры поля.

Внутри `BusinessMetadataRegistry` добавьте:

```java
private record SqlFieldSignature(
        String jsonName,
        FieldKind kind
) {
}
```

Затем замените `resolveGlobalCandidate` на такой вариант:

```java
private static FieldMetadata resolveGlobalCandidate(
        String fieldKey,
        List<FieldMetadata> candidates
) {
    if (candidates == null || candidates.isEmpty()) {
        throw new IllegalArgumentException("No candidates for fieldKey '" + fieldKey + "'");
    }

    Map<SqlFieldSignature, List<FieldMetadata>> bySqlSignature = candidates.stream()
            .collect(Collectors.groupingBy(
                    BusinessMetadataRegistry::sqlFieldSignature,
                    LinkedHashMap::new,
                    Collectors.toList()
            ));

    if (bySqlSignature.size() == 1) {
        return bySqlSignature.values()
                .iterator()
                .next()
                .get(0);
    }

    String candidatesDescription = candidates.stream()
            .map(BusinessMetadataRegistry::describeCandidate)
            .distinct()
            .collect(Collectors.joining("; "));

    throw new IllegalArgumentException(
            "Ambiguous fieldKey '" + fieldKey + "'. " +
                    "Found fields with different SQL definitions: " +
                    candidatesDescription
    );
}
```

И добавьте метод:

```java
private static SqlFieldSignature sqlFieldSignature(FieldMetadata field) {
    return new SqlFieldSignature(
            field.jsonName(),
            field.kind()
    );
}
```

Метод `describeCandidate`:

```java
private static String describeCandidate(FieldMetadata field) {
    return field.declaredInClass() +
            "." +
            field.name() +
            " jsonName=" +
            field.jsonName() +
            " kind=" +
            field.kind() +
            " rawType=" +
            field.rawType() +
            " restrictions=" +
            field.field().restrictions();
}
```

# Что будет после исправления

Для поля:

```json
{
  "fieldKey": "side"
}
```

если в metadata есть:

```text
Trade.side          jsonName=side type=STRING
TradeSaveEvent.side jsonName=side type=STRING
```

валидатор выберет первое совместимое поле и SQL будет строиться так:

```sql
"side" AS "side"
```

А ошибка `AMBIGUOUS_FIELD` больше не появится.

---

## Важное замечание про `restrictions`

Если у `Trade.side` и `TradeSaveEvent.side` разные `restrictions`, например:

```text
Trade.side          ENUM [BUY, SELL]
TradeSaveEvent.side ENUM [PAY, RECEIVE]
```

то после быстрого фикса валидатор возьмет `restrictions` от первого найденного поля.

Если вам важно валидировать `restrictions` максимально строго, лучше следующим шагом сделать отдельную логику объединения restrictions:

- для `ENUM` — объединять допустимые значения;
- для `MAX_LENGTH` — брать максимальное или минимальное значение в зависимости от вашей бизнес-логики.

Но для текущей ошибки достаточно считать поля одинаковыми, если у них совпадают `jsonName` и тип.

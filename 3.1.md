Отличное уточнение! Тогда проблема совсем в другом.

## Новая картина

Ты **уже загрузил страницу** (HTTP GET прошёл), закрыл браузер, открыл снова — и WebSocket снова не работает пока не сделаешь HTTP GET. Это означает что проблема **не в инициализации Spring**, а в **сетевом уровне Kubernetes**.

## Главный подозреваемый — Session Affinity / балансировка

В Kubernetes каждый раз когда ты открываешь новую вкладку/браузер — твой запрос может попасть на **другой под**. При этом:

```
Браузер → Ingress → Pod A  (HTTP GET, всё ок)
Браузер → Ingress → Pod B  (WebSocket handshake — проблема)
```

Но это тоже не объясняет полностью...

## Скорее всего проблема в Ingress — WebSocket не проксируется правильно

Самая частая причина именно такого поведения — **Ingress не передаёт заголовки Upgrade/Connection** для WebSocket:

```yaml
# Для nginx-ingress обязательно нужны эти аннотации:
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-http-version: "1.1"  # ключевое!
    nginx.ingress.kubernetes.io/configuration-snippet: |
      proxy_set_header Upgrade $http_upgrade;
      proxy_set_header Connection "upgrade";
```

## Почему HTTP GET "лечит" проблему?

Есть интересная гипотеза — **keep-alive соединение**. Когда браузер делает HTTP GET:
1. Устанавливается TCP соединение с подом
2. Браузер держит его как keep-alive
3. WebSocket Upgrade происходит **по уже существующему соединению** минуя проблемный путь через Ingress

При новом открытии браузера — новое TCP соединение, которое снова идёт через Ingress и снова ломается.

Нет, проблема **почти точно не в `internalArrayAlias(...)`**.

# Исправь `PivotQueryResponseFactory.computeAggregate(...)`

Лучше не использовать `double` для денежных/decimal-значений. Используй `BigDecimal`.

Добавь imports:

```java
import java.math.BigDecimal;
import java.math.RoundingMode;
```

Замени метод `computeAggregate(...)` на такой:

```java
@SuppressWarnings({"unchecked", "rawtypes"})
private Object computeAggregate(List<Object> values, String aggFunc) {
    if (values == null || values.isEmpty()) {
        return null;
    }

    List<Object> nonNullValues = values.stream()
            .filter(Objects::nonNull)
            .toList();

    if (nonNullValues.isEmpty()) {
        return null;
    }

    return switch (aggFunc.toLowerCase(Locale.ROOT)) {
        case "count" -> nonNullValues.size();

        case "sum" -> nonNullValues.stream()
                .map(this::toBigDecimal)
                .filter(Objects::nonNull)
                .reduce(BigDecimal.ZERO, BigDecimal::add);

        case "avg" -> {
            List<BigDecimal> decimals = nonNullValues.stream()
                    .map(this::toBigDecimal)
                    .filter(Objects::nonNull)
                    .toList();

            if (decimals.isEmpty()) {
                yield null;
            }

            BigDecimal sum = decimals.stream()
                    .reduce(BigDecimal.ZERO, BigDecimal::add);

            yield sum.divide(
                    BigDecimal.valueOf(decimals.size()),
                    6,
                    RoundingMode.HALF_UP
            );
        }

        case "min" -> nonNullValues.stream()
                .min(this::compareValues)
                .orElse(null);

        case "max" -> nonNullValues.stream()
                .max(this::compareValues)
                .orElse(null);

        default -> null;
    };
}
```

И добавь helper-методы:

```java
private BigDecimal toBigDecimal(Object value) {
    if (value == null) {
        return null;
    }

    if (value instanceof BigDecimal decimal) {
        return decimal;
    }

    if (value instanceof Number number) {
        return new BigDecimal(number.toString());
    }

    if (value instanceof CharSequence sequence) {
        String text = sequence.toString().trim();

        if (text.isEmpty()) {
            return null;
        }

        return new BigDecimal(text);
    }

    throw new IllegalArgumentException(
            "Cannot convert value to BigDecimal: " +
                    value +
                    ", class=" +
                    value.getClass().getName()
    );
}

@SuppressWarnings({"rawtypes", "unchecked"})
private int compareValues(Object left, Object right) {
    BigDecimal leftDecimal = tryToBigDecimal(left);
    BigDecimal rightDecimal = tryToBigDecimal(right);

    if (leftDecimal != null && rightDecimal != null) {
        return leftDecimal.compareTo(rightDecimal);
    }

    if (left instanceof Comparable leftComparable
            && right != null
            && left.getClass().isAssignableFrom(right.getClass())) {
        return leftComparable.compareTo(right);
    }

    return String.valueOf(left).compareTo(String.valueOf(right));
}

private BigDecimal tryToBigDecimal(Object value) {
    try {
        return toBigDecimal(value);
    } catch (Exception e) {
        return null;
    }
}
```

# `internalArrayAlias(...)` лучше вернуть обратно с `__aj_`

Текущий метод нормальный:

```java
private String internalArrayAlias(
        String columnName,
        Map<String, String> existingAliases
) {
    String base = "__aj_" + columnName.replaceAll("[^A-Za-z0-9_]", "_");

    String alias = base;
    int index = 2;

    while (existingAliases.containsValue(alias)) {
        alias = base + "_" + index;
        index++;
    }

    return alias;
}
```

Я бы оставил префикс `__aj_`.

То есть лучше, чтобы SQL был таким:

```sql
ARRAY JOIN
    "cashflows.direction" AS "__aj_cashflows_direction",
    "cashflows.paymentAmount" AS "__aj_cashflows_paymentAmount"
```

А не таким:

```sql
ARRAY JOIN
    "cashflows.direction" AS "cashflows_direction",
    "cashflows.paymentAmount" AS "cashflows_paymentAmount"
```

Префикс полезен, потому что это технические alias-ы. Так меньше шанс пересечься с реальными колонками.

# Но в SQL есть ещё один момент

Сейчас `ARRAY JOIN` строится по **всем array-полям из `columns`**, хотя в итоговом grouped/pivot SQL используются только:

```text
Trade.cashflows.direction
Trade.cashflows.paymentCurrency.code
Trade.cashflows.paymentAmount
```

Но в `ARRAY JOIN` попали ещё:

```text
cashflows.paymentDate
cashflows.counterparty_code
cashflows.state
cashflows.type
cashflows.periodStart
cashflows.periodEnd
cashflows.notionalAmount
cashflows.originalCurrency_code
cashflows.conversionDate
cashflows.conversionSource_alias
cashflows.conversionRate
```

Это не причина текущей Java-ошибки, но лучше оптимизировать.

Для pivot/group query в `ARRAY JOIN` должны попадать только поля, которые реально участвуют в SQL:

- `firstRowGroupColumns`;
- `pivotColumns`;
- `valueColumns`;
- `filterModel`;
- `groupKeys`;
- `sortModel`.

А не все `columns`.

Иначе:

1. запрос тяжелее;
2. если длина какого-то массива отличается, ClickHouse может вернуть ошибку;
3. primary array column может выбраться неудачно.

# Быстрый фикс для `buildSqlContext`

Сейчас:

```java
addFieldKeys(config.path("columns"), result);
```

из-за этого в `ARRAY JOIN` попадает много лишних массивных колонок.

Лучше сделать так: если есть группировка/pivot/aggregation, не добавлять обычные `columns` в array join.

```java
private Set<String> collectAllFieldKeys(JsonNode config) {
    Set<String> result = new LinkedHashSet<>();

    boolean hasRowGroups = config.path("rowGroupColumns").isArray()
            && !config.path("rowGroupColumns").isEmpty();

    boolean hasPivot = config.path("pivotColumns").isArray()
            && !config.path("pivotColumns").isEmpty();

    boolean hasValues = config.path("valueColumns").isArray()
            && !config.path("valueColumns").isEmpty();

    boolean relationalQuery = hasRowGroups || hasPivot || hasValues;

    if (!relationalQuery) {
        addFieldKeys(config.path("columns"), result);
    }

    addFieldKeys(config.path("rowGroupColumns"), result);
    addFieldKeys(config.path("pivotColumns"), result);
    addFieldKeys(config.path("valueColumns"), result);
    addFieldKeys(config.path("groupKeys"), result);

    JsonNode sortModel = config.path("sortModel");

    if (sortModel.isArray()) {
        for (JsonNode sortItem : sortModel) {
            String colId = sortItem.path("colId").asText(null);

            if (colId != null && !colId.isBlank()) {
                result.add(colId);
            }
        }
    }

    JsonNode filterModel = config.path("filterModel");

    if (filterModel != null && filterModel.isObject()) {
        filterModel.fieldNames().forEachRemaining(result::add);
    }

    return result;
}
```

Тогда для твоего запроса `ARRAY JOIN` будет компактнее:

```sql
ARRAY JOIN
    arrayEnumerate("__some_primary_array__") AS "__arrayIndex",
    "cashflows.direction" AS "__aj_cashflows_direction",
    "cashflows.paymentCurrency_code" AS "__aj_cashflows_paymentCurrency_code",
    "cashflows.paymentAmount" AS "__aj_cashflows_paymentAmount"
```

# Ещё лучше выбрать primary array column из valueColumns

Сейчас primary column выбирается первой из `LinkedHashMap`:

```java
String primaryArrayColumn = arrayFieldsByColumn.values()
        .iterator()
        .next()
        .columnName();
```

В твоём SQL получилось:

```sql
arrayEnumerate("cashflows.paymentDate") AS "__arrayIndex"
```

Хотя `paymentDate` вообще не участвует в SELECT/GROUP BY в текущем SQL.

Лучше primary array column выбирать так:

1. первое array-поле из `valueColumns`;
2. если нет — первое из `pivotColumns`;
3. если нет — первое из `rowGroupColumns`;
4. если нет — любое.

Но это уже улучшение, не причина текущего `String cannot be cast to Number`.
